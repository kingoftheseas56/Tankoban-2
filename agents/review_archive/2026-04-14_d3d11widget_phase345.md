### Gaps (Missing or Simplified)

Ranked P0 / P1 / P2.

**P0:** None.

**P1:** None.

**P2:**

- **Phase 5 pending-import slot has no memory-order guarantee.** `attachD3D11Texture` at `D3D11Widget.cpp:852-862` writes `m_pendingD3DHandle` / `m_pendingD3DWidth` / `m_pendingD3DHeight` / `m_d3dActive` from the calling thread; `processPendingImport` at `:674-744` reads them on the render-timer thread. The code comment at `:852-856` says "callable from any thread," but the members are plain `quintptr` / `int` / `bool` — no atomics, no memory-order fence, no mutex. In practice VideoPlayer's Qt signal/slot plumbing delivers `attachD3D11Texture` on the widget's thread (main thread), and the render timer also fires on the widget's thread, so the race is formal-not-actual. Fix options: (a) tighten the comment to "must be called on the widget's thread" — which is the de facto invariant — or (b) wrap the slot in a tiny critical section / use `std::atomic<quintptr>` for the handle so the "any thread" claim holds up. Either fine; pick to match intent.
- **Phase 4 `detachShm` resets `m_videoTexW/H = 0` but keeps the texture alive.** `D3D11Widget.cpp:839-843`. Next `attachShm` + `consumeShmFrame` sees `f.width != m_videoTexW` (since m_videoTexW is 0) → always recreates the texture on re-attach even if dims are unchanged. One wasted allocation per attach/detach cycle. Cosmetic — video attach/detach is infrequent. Fix: leave `m_videoTexW/H` at their prior values, or release the texture too so the `m_videoTexture == nullptr` branch does the allocation naturally.
- **Phase 5 `processPendingImport` defensive pre-release** at `D3D11Widget.cpp:728-729`. Comment says "should be null after detach" but the code releases anyway. Correct belt-and-braces. Flagging only because it's the kind of thing that can mask a bug if someone later introduces a re-entrant attach without detach — the defensive release will quietly paper over a leaked previous import. Could add a qWarning when the defensive branch fires to surface unexpected state.
- **Phase 5 cross-device synchronization is implicit.** The sidecar writes to the shared texture via `CopySubresourceRegion + Flush` on its side; our side reads via `processPendingImport` → `OpenSharedResource1` then `PSSetShaderResources`. D3D11 shared NT handles don't auto-synchronize — if the sidecar's source texture was created with `D3D11_RESOURCE_MISC_SHARED_KEYEDMUTEX`, the consumer must `AcquireSync` / `ReleaseSync` around reads. If it was created with plain `D3D11_RESOURCE_MISC_SHARED_NTHANDLE`, the producer's `Flush` is best-effort and tearing is possible under producer/consumer race. D3D11Widget does neither — implicit-sync assumption. Flagging because this only matters at Phase 7.1 when real sidecar frames start flowing; worth confirming with Agent 3 that the sidecar creates the shared texture without keyed mutex (so `Flush` on the producer + implicit D3D11 serialization is the chosen model). See Question 3.
- **`consumeShmFrame` silently drops `readLatest()` invalid frames.** `D3D11Widget.cpp:753-754` returns `false` without logging. Under normal operation this is expected (ring buffer may return stale slots); under sidecar starvation it's the same code path and the user sees a frozen frame. Matches `FrameCanvas` behavior so not a regression, but future debug would benefit from a rate-limited qDebug ("N frames dropped in last M ms") for observability. Polish.
- **`uploadColorParams` `Map` failure silently returns.** `D3D11Widget.cpp:818-822`. Comment correctly explains "per-frame, log spam". Accepted, but a one-time flag-and-log (warn once, then suppress) would still catch the rare-but-real case where device removal silently breaks the cbuffer update. Minor polish.
- **`D3D11Widget::attachD3D11Texture` calls `update()` but render is timer-driven.** `D3D11Widget.cpp:861` — the `update()` call is harmless (with `WA_PaintOnScreen` set, `paintEvent` is a no-op anyway) and mirrors a FrameCanvas pattern, but it's cargo-culted in this context: the 16 ms render timer will tick within 16 ms of the attach regardless. Harmless; noting because a future reader might be confused.

### Answers to Agent 3's scrutiny points

1. **HLSL math 1:1 with `video.frag`.** Verified line-for-line above — no semantic drift, all idiom swaps (`vec3→float3`, `mix→lerp`, `clamp(0,1)→saturate`, scalar auto-broadcast in `pow`) are correct. Hemanth's visual identity-transform verification at default params is the strongest possible confirmation.
2. **Phase 4 USAGE_DEFAULT + UpdateSubresource.** Correct choice for full-rewrite-per-frame BGRA. DYNAMIC + `Map(WRITE_DISCARD)` would add an extra copy layer without functional benefit. Matches FrameCanvas's effective behavior via QRhi.
3. **SRV priority chain fallback on Phase 5 detach.** Correctly falls back to the SHM path when `m_d3dActive` clears — `consumeShmFrame` needs `m_polling` to still be set by VideoPlayer, but if polling is off (legitimately, because the user stopped playback), the test checker is the correct terminal state. See Question 2 for the Phase 7 wiring concern.
4. **`zeroCopyActivated` signal threading.** Qt's default `AutoConnection` is safe: since `processPendingImport` runs on the widget's thread (render timer is a QTimer owned by D3D11Widget, ticks on the widget's thread = main thread), and VideoPlayer lives on the same thread, `AutoConnection` resolves to `DirectConnection` and the signal dispatches synchronously. No `Qt::QueuedConnection` annotation needed. If VideoPlayer ever moves to a worker thread in a future refactor, the connection type re-resolves automatically.
5. **Phase 5 format inference via `GetDesc`.** Sound, not premature. One-line cost, removes a hardcoded BGRA8 assumption, correctly positions Phase B+ HDR work (10-bit `DXGI_FORMAT_R10G10B10A2_UNORM` etc.) to drop in without shader changes. Good call.

### Questions for Agent 3

1. **Thread-safety claim on `attachD3D11Texture`** (P2 #1). Is the "callable from any thread" comment aspirational, or is there a real plan to call it from a worker thread? If the answer is "main thread only in practice," tighten the comment; if "any thread," add `std::atomic<quintptr>` on the handle slot. Not gating the verdict — pick whichever matches intent.
2. **Phase 7 wiring: detach-d3d11 → polling re-arm.** When VideoPlayer detaches the D3D11 shared texture to fall back to SHM (e.g., GPU change, sidecar restart), does VideoPlayer also re-issue `startPolling()`? If not, the SRV priority chain falls through to the test checker, not the SHM video path. Worth a Phase 7.1 integration-test item.
3. **Cross-process shared-texture synchronization model** (P2 #4). Does the sidecar create the shared texture with `D3D11_RESOURCE_MISC_SHARED_KEYEDMUTEX` (producer/consumer with explicit sync) or plain `D3D11_RESOURCE_MISC_SHARED_NTHANDLE` (implicit-sync assumption)? If the latter, the `Flush`-on-producer model is OK under normal cadence but can tear under race. If the former, D3D11Widget needs `AcquireSync`/`ReleaseSync` around the Draw call — not shipped in Phase 5. Answer determines whether a follow-up item is needed.
