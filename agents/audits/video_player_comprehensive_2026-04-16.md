## Symptom 1 — Slow Video Startup / Blank First Frame

### Observations

Our side:

- `VideoPlayer::openFile` tears down the current UI, sets the new file identity, and either starts the sidecar or sends a stop command before sending `open` for the new file (`src/ui/player/VideoPlayer.cpp:263`, `src/ui/player/VideoPlayer.cpp:281`, `src/ui/player/VideoPlayer.cpp:347`, `src/ui/player/VideoPlayer.cpp:371`, `src/ui/player/VideoPlayer.cpp:383`). It does not set a visible loading/buffering HUD state in this path; it ends by calling `showControls()` (`src/ui/player/VideoPlayer.cpp:388`).
- `SidecarProcess::start` starts the native sidecar and waits synchronously up to 5000 ms for process startup (`src/ui/player/SidecarProcess.cpp:98`, `src/ui/player/SidecarProcess.cpp:108`). That is a bounded UI-thread wait on cold start before any decoder work is visible.
- Sidecar `handle_open` acknowledges the command, sets state to `OPEN_PENDING`, emits `state_changed` with `"opening"`, and then spawns `open_worker` (`native_sidecar/src/main.cpp:644`, `native_sidecar/src/main.cpp:655`, `native_sidecar/src/main.cpp:659`). Current `VideoPlayer` connections use sidecar `ready`, `firstFrame`, `timeUpdate`, and `tracksChanged` (`src/ui/player/VideoPlayer.cpp:204`), but I did not find a visible loading indicator bound to the `"opening"` state.
- `open_worker` performs `probe_file(path)` before creating decoders and before starting the periodic time-update thread (`native_sidecar/src/main.cpp:256`, `native_sidecar/src/main.cpp:272`, `native_sidecar/src/main.cpp:639`). Track and media payloads are built before decode, but the first `tracks_changed` and `media_info` emissions are inside the first-frame event callback (`native_sidecar/src/main.cpp:309`, `native_sidecar/src/main.cpp:346`, `native_sidecar/src/main.cpp:392`, `native_sidecar/src/main.cpp:402`, `native_sidecar/src/main.cpp:418`).

Audio decoder startup also opens its own FFmpeg input and calls `avformat_find_stream_info` (`native_sidecar/src/audio_decoder.cpp:182`, `native_sidecar/src/audio_decoder.cpp:199`). For URLs, both decoders set network-related timeout/probe options, including `rw_timeout=30000000` and large probe/analyze values (`native_sidecar/src/video_decoder.cpp:168`, `native_sidecar/src/video_decoder.cpp:179`, `native_sidecar/src/audio_decoder.cpp:172`, `native_sidecar/src/audio_decoder.cpp:180`).
- The D3D11 path has waitable-swapchain support after the recent perf work: `FrameCanvas` creates the swapchain with `DXGI_SWAP_CHAIN_FLAG_FRAME_LATENCY_WAITABLE_OBJECT` and starts a waitable render loop if available (`src/ui/player/FrameCanvas.cpp:226`, `src/ui/player/FrameCanvas.cpp:235`, `src/ui/player/FrameCanvas.cpp:315`, `src/ui/player/FrameCanvas.cpp:320`, `src/ui/player/FrameCanvas.cpp:444`, `src/ui/player/FrameCanvas.cpp:467`, `src/ui/player/FrameCanvas.cpp:477`). Microsoft documents that this flag creates a waitable object to avoid rendering while a frame is still being presented, and that frame latency should be controlled through `IDXGISwapChain2::SetMaximumFrameLatency` for such swapchains (Microsoft DXGI flag docs; Microsoft `SetMaximumFrameLatency` docs).
- `FrameCanvas::renderFrame` clears the render target to black and then calls `drawTexturedQuad` (`src/ui/player/FrameCanvas.cpp:676`, `src/ui/player/FrameCanvas.cpp:678`). `drawTexturedQuad` returns without drawing video when no texture/SRV is available (`src/ui/player/FrameCanvas.cpp:849`, `src/ui/player/FrameCanvas.cpp:869`). So a blank black frame is the expected canvas presentation before a decoded or shared texture exists.
- The fast D3D11 subtitle-overlay path emits `first_frame` only after a video frame has been copied/imported and the clock path has allowed display (`native_sidecar/src/video_decoder.cpp:478`, `native_sidecar/src/video_decoder.cpp:489`, `native_sidecar/src/video_decoder.cpp:500`, `native_sidecar/src/video_decoder.cpp:577`). The slower path has a clock-start wait with a 10 second deadline (`native_sidecar/src/video_decoder.cpp:769`, `native_sidecar/src/video_decoder.cpp:789`) before first-frame emission (`native_sidecar/src/video_decoder.cpp:904`).

- `native_sidecar/src/main.cpp` joins any existing open thread in teardown/open/stop paths (`native_sidecar/src/main.cpp:178`, `native_sidecar/src/main.cpp:179`, `native_sidecar/src/main.cpp:644`, `native_sidecar/src/main.cpp:649`, `native_sidecar/src/main.cpp:666`, `native_sidecar/src/main.cpp:668`). If an open worker is inside a slow probe/open/read path, later stop/open sequencing waits for that worker to finish before the next visible state can progress.

### Hypotheses

- Hypothesis — Agent 3 to validate: The reported 30 second blank-screen cases are likely dominated by sidecar/network/decode readiness rather than the recent DXGI present cadence, because the current code can retry stalled reads for about 30 seconds while the UI has no handled buffering event and the canvas only has black clear before first texture.
- Hypothesis — Agent 3 to validate: Cold-start blank delay may have a smaller but visible UI-thread component from `waitForStarted(5000)`, followed by asynchronous probe/open/decode work. The relative contribution likely differs between cold sidecar launch, warm local files, and network URLs.
- Hypothesis — Agent 3 to validate: Metadata and first frame are currently coupled too tightly, because `tracks_changed` and `media_info` are emitted from the first-frame callback even though track and probe data exist earlier in `open_worker`. - Hypothesis — Agent 3 to validate: Stop/open transitions can inherit latency from an in-flight open worker because teardown joins that worker. This may show up most often when the user switches videos during a slow network/probe path.

P1:

- Stale HUD duration/timestamps/tracks during video switch. This is highly visible and undermines trust in the player, but it is bounded to transition windows unless it persists after events arrive.

- Tracks/EQ/Filters chip polish and feature depth relative to IINA/QMPlay2/VLC. These are important for player identity and perceived quality, but less blocking than blank startup, subtitle correctness, or stale core playback metadata.
