## Observed behavior (in our codebase)

### Stream startup and buffer gate

- Magnet streams are added to libtorrent in non-paused mode, then `StreamEngine` enables sequential download for the torrent via `m_torrentEngine->setSequentialDownload(addedHash, true)`. See `src/core/stream/StreamEngine.cpp:136` and `src/core/stream/StreamEngine.cpp:147`, plus `src/core/torrent/TorrentEngine.cpp:472`.
- Startup progress is derived from selected-file total progress. `StreamEngine` reads `torrentFiles()`, multiplies the selected file's `progress` by `totalSize`, stores that as `downloadedBytes`, and exposes `fileProgress = downloaded / totalSize`. See `src/core/stream/StreamEngine.cpp:185`, `src/core/stream/StreamEngine.cpp:193`, and `src/core/stream/StreamEngine.cpp:204`.
- The actual startup gate is not total-file progress. It is a fixed first-byte contiguous gate: the first `2 * 1024 * 1024` bytes from selected-file offset 0 must map to completed pieces before playback is marked ready. See `src/core/stream/StreamEngine.cpp:206` and `src/core/stream/StreamEngine.cpp:211`.
- When that 2 MB head gate has not passed, the returned error is `FILE_NOT_READY` with `"Buffering... %1%"`, where `%1` is computed from total selected-file progress. See `src/core/stream/StreamEngine.cpp:211` and `src/core/stream/StreamEngine.cpp:215`.
- `StreamPlayerController` repeats the same total-file metric in the UI: `pct = result.fileProgress * 100`, `mbDownloaded = result.downloadedBytes`, then emits `bufferUpdate(statusText, result.fileProgress * 100.0)`. See `src/ui/pages/stream/StreamPlayerController.cpp:125`, `src/ui/pages/stream/StreamPlayerController.cpp:130`, and `src/ui/pages/stream/StreamPlayerController.cpp:155`.

### Piece selection and priority

- File-level priority is applied after metadata/file selection: all files receive priority 0, and the selected file receives priority 7 through `TorrentEngine::setFilePriorities()`. See `src/core/stream/StreamEngine.cpp:354`, `src/core/stream/StreamEngine.cpp:456`, and `src/core/torrent/TorrentEngine.cpp:458`.
- The playback path does not expose a piece-priority or deadline API. The torrent wrapper shown here has `prioritize_files()` and `set_flags(lt::torrent_flags::sequential_download)`, but no observed use of `set_piece_deadline()`, `prioritize_pieces()`, or `piece_priority()` in the inspected stream/torrent files. See `src/core/torrent/TorrentEngine.cpp:458` and `src/core/torrent/TorrentEngine.cpp:472`.
- `haveContiguousBytes()` maps a file byte span to a first/last piece range and returns true only when every mapped piece is complete according to `handle.have_piece()`. See `src/core/torrent/TorrentEngine.cpp:737`, `src/core/torrent/TorrentEngine.cpp:753`, and `src/core/torrent/TorrentEngine.cpp:759`.

### File-vs-HTTP serving

- `StreamEngine.h` still describes torrent-backed playback as local HTTP: `StreamPlaybackMode::LocalHttp` is "torrent-backed (magnet) path served via local HTTP", and `StreamFileResult::url` says it is a `http://127.0.0.1:{port}/stream/{hash}/{fileIndex}` URL or direct URL. See `src/core/stream/StreamEngine.h:14`, `src/core/stream/StreamEngine.h:21`, and `src/core/stream/StreamEngine.h:54`.
- `StreamEngine.cpp` builds a local HTTP URL in `buildStreamUrl()`, but the normal magnet playback return path bypasses it. After the 2 MB gate passes, the implementation explicitly returns a direct file path instead of an HTTP URL. See `src/core/stream/StreamEngine.cpp:446`, `src/core/stream/StreamEngine.cpp:219`, and `src/core/stream/StreamEngine.cpp:244`.
- The direct-file return includes a comment acknowledging sparse-zero risk: `have_piece()` may be true while bytes are still in libtorrent's memory cache, causing external readers to see sparse zeros. The code calls `flushCache()` before returning the file path. See `src/core/stream/StreamEngine.cpp:224` and `src/core/torrent/TorrentEngine.cpp:766`.
- The HTTP server has a piece-aware wait loop with 256 KiB chunks and a 15-second timeout. See `src/core/stream/StreamHttpServer.cpp:68` and `src/core/stream/StreamHttpServer.cpp:74`.
- If the HTTP server is used, it supports byte ranges and sends `206 Partial Content`, `Content-Range`, `Content-Length`, and `Accept-Ranges: bytes`. See `src/core/stream/StreamHttpServer.cpp:140`, `src/core/stream/StreamHttpServer.cpp:168`, and `src/core/stream/StreamHttpServer.cpp:178`.
- The HTTP server calls `waitForPieces()` before reading each chunk, but it ignores the boolean result. If the 15-second wait expires, the code still reads from the file and writes the chunk. See `src/core/stream/StreamHttpServer.cpp:203`, `src/core/stream/StreamHttpServer.cpp:207`, and `src/core/stream/StreamHttpServer.cpp:210`.

### Decoder launch, resume, and seek

- `StreamPage::onReadyToPlay()` passes the URL/path returned by `StreamPlayerController` to `VideoPlayer::openFile()`. The local variable is named `httpUrl`, but current `StreamEngine` may supply a local file path. See `src/ui/pages/StreamPage.cpp:1162` and `src/ui/pages/StreamPage.cpp:1277`.
- Stream-domain progress is read before launch, and a saved offset is passed as `streamResumeSec` if the saved position is greater than 2 seconds and less than 95 percent of duration. See `src/ui/pages/StreamPage.cpp:1267`, `src/ui/pages/StreamPage.cpp:1270`, and `src/ui/pages/StreamPage.cpp:1277`.
- `VideoPlayer::openFile()` records caller-supplied `startPositionSec` in `m_pendingStartSec`. See `src/ui/player/VideoPlayer.cpp:312` and `src/ui/player/VideoPlayer.cpp:313`.
- If the sidecar is already running, `VideoPlayer::openFile()` sends `m_sidecar->sendOpen(filePath)` without passing `m_pendingStartSec`. If the sidecar is not running, `onSidecarReady()` sends `m_pendingFile` with `m_pendingStartSec`. See `src/ui/player/VideoPlayer.cpp:325` and `src/ui/player/VideoPlayer.cpp:397`.
- The sidecar probe path opens the path directly with ffmpeg and a 5 MB probe size. See `native_sidecar/src/demuxer.cpp:15` and `native_sidecar/src/demuxer.cpp:21`.
- `VideoDecoder` enables HTTP-specific reconnect, timeout, larger probe, and retry behavior only when the path starts with `http://` or `https://`. See `native_sidecar/src/video_decoder.cpp:166`, `native_sidecar/src/video_decoder.cpp:168`, and `native_sidecar/src/video_decoder.cpp:175`.
- Initial resume seek is executed inside ffmpeg through `av_seek_frame()` before decoding starts. See `native_sidecar/src/video_decoder.cpp:322` and `native_sidecar/src/video_decoder.cpp:325`.
- Mid-playback `av_read_frame()` failures receive special buffering/retry handling only in HTTP mode. Non-HTTP failures emit `DECODE_FAILED` and break. See `native_sidecar/src/video_decoder.cpp:807`, `native_sidecar/src/video_decoder.cpp:834`, and `native_sidecar/src/video_decoder.cpp:850`.

### Session settings and recovery

- Torrent session settings are general download settings: DHT/LSD/NATPMP/UPNP enabled, 200 connections, 5 active downloads, request timeout 10 seconds, peer timeout 20 seconds, and tracker announce fanout. See `src/core/torrent/TorrentEngine.cpp:169`, `src/core/torrent/TorrentEngine.cpp:192`, and `src/core/torrent/TorrentEngine.cpp:199`.
- The settings block does not show streaming-specific disk/read-cache tuning or time-critical piece configuration in the inspected code. The code comments call the mode "less aggressive than streaming". See `src/core/torrent/TorrentEngine.cpp:194`.
- On launch failure, `StreamPage::onStreamFailed()` resets stream state and shows `Stream failed: <message>` for 3 seconds before returning to browse if the controller is inactive. See `src/ui/pages/StreamPage.cpp:1290` and `src/ui/pages/StreamPage.cpp:1302`.
- There is no observed playback-frontier feedback loop from the sidecar to `TorrentEngine` that reprioritizes pieces near the current decoder position or near a user/resume seek target.

## Reference behavior

### libtorrent streaming guidance

- libtorrent distinguishes `sequential_download` from its time-critical streaming logic. It says `sequential_download` keeps queues full and pieces arrive roughly in order, but it is sub-optimal for streaming because an early piece may be requested from a slow peer while faster peers request later pieces. Source: https://www.libtorrent.org/streaming.html.
- The same guide describes time-critical pieces as the ones the player will need next, with peers sorted by estimated queue time and urgent pieces requested from the peer most likely to deliver soonest. Source: https://www.libtorrent.org/streaming.html.
- libtorrent exposes `torrent_handle::set_piece_deadline()`, `clear_piece_deadlines()`, and `reset_piece_deadline()`. The reference says pieces with nearer deadlines are prioritized over pieces with later deadlines, and deadlines can be changed by calling the function again. Source: https://libtorrent.org/reference-Torrent_Handle.html.
- libtorrent documents `have_piece()` as true only when the piece has been completely downloaded and written to disk. Source: https://libtorrent.org/reference-Torrent_Handle.html.

### torrent-stream

- torrent-stream's package description is "Low level streaming torrent client that exposes files as node.js streams and downloads pieces based on demand." Source: https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/package.json#L4.
- `file.createReadStream()` creates a `fileStream`, then selects that stream's start/end pieces at high priority and deselects them when the stream closes. Source: https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/index.js#L172.
- `FileStream` computes start/end pieces from the requested byte range and defines a small critical window: up to roughly 1 MB or 2 pieces. If the current piece is not present, `notify()` marks that piece range as critical instead of reading from the file. Source: https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/lib/file-stream.js#L12 and https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/lib/file-stream.js#L24.
- The torrent-stream picker iterates selections in priority order and requests selected/critical pieces from peers; priority selections can be shuffled so high-priority streams continue to get request slots. Source: https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/index.js#L390 and https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/index.js#L639.

### Peerflix

- Peerflix builds an HTTP server over torrent-stream. It selects a default file, exposes an HTTP URL, and handles HTTP requests through the selected torrent file. Source: https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L30 and https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L44.
- Peerflix parses the `Range` header, sends `Accept-Ranges`, and for partial requests returns `206` plus `Content-Range`; the response body is `pump(file.createReadStream(range), response)`. Source: https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L150 and https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L164.
- Because Peerflix uses torrent-stream's `createReadStream(range)`, range reads are tied to torrent-stream's demand-driven selection/critical-piece behavior rather than direct sparse-file reads. Source: https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L168 and https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/lib/file-stream.js#L36.

### Webtor

- Webtor torrent-web-seeder serves cached files directly when complete, but falls back to a torrent reader when cache/vault cannot satisfy the request. It then calls `http.ServeContent()` with that reader, not a bare sparse file. Source: https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L196 and https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L221.
- The Webtor torrent reader is created from `f.NewReader()`, then set to responsive mode and given a bounded readahead function before it is returned to `ServeContent()`. Source: https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L287 and https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L295.
- Webtor's `PieceReader` delegates reads and seeks to a torrent reader positioned relative to a torrent piece, again keeping reads inside a torrent-aware userspace layer. Source: https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/piece_reader.go#L18 and https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/piece_reader.go#L26.
- Webtor content-transcoder probes source URLs through ffprobe, creates HLS sessions, and on seek restarts/injects ffmpeg seek parameters into the transcoding run. Source: https://github.com/webtor-io/content-transcoder/blob/90b9c9f2a700a4a78325137b815bab7fc45532f2/services/content_prober.go#L148, https://github.com/webtor-io/content-transcoder/blob/90b9c9f2a700a4a78325137b815bab7fc45532f2/services/web.go#L264, and https://github.com/webtor-io/content-transcoder/blob/90b9c9f2a700a4a78325137b815bab7fc45532f2/services/transcode_run.go#L117.

## Gaps (ranked P0 / P1 / P2)

### P0

- P0 - Normal magnet playback exposes a sparse, partially-downloaded local file directly to ffmpeg after only the first 2 MB are contiguous. Our hot path returns `filePath` (`src/core/stream/StreamEngine.cpp:219`, `src/core/stream/StreamEngine.cpp:244`), while Peerflix serves HTTP ranges through `file.createReadStream(range)` (https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L168), torrent-stream blocks reads behind a demand-driven critical-piece stream (https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/lib/file-stream.js#L36), and Webtor serves through a torrent reader (`https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L295`). This is the largest concrete mismatch for the reported random forward jumps: our decoder can read beyond the downloaded frontier as a plain file, while the references keep the read path torrent-aware.
- P0 - The active torrent strategy uses file priority plus `sequential_download`, with no observed playback-frontier piece deadlines or sliding high-priority window. Our code sets selected-file priority and sequential mode (`src/core/stream/StreamEngine.cpp:456`, `src/core/torrent/TorrentEngine.cpp:478`); libtorrent says `sequential_download` is sub-optimal for streaming and documents deadline-based priority (`https://www.libtorrent.org/streaming.html`, `https://libtorrent.org/reference-Torrent_Handle.html`); torrent-stream selects the requested range and marks missing current pieces critical (`https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/index.js#L176`, `https://github.com/mafintosh/torrent-stream/blob/adc53d6ced42959ae37553e9f11aa6372e485ca0/lib/file-stream.js#L38`). This is a core search-to-play degradation because ffmpeg read-ahead is not tied to torrent request priority.

### P1

- P1 - The buffer indicator describes whole-file progress while the actual readiness gate is "first 2 MB contiguous from byte 0." The displayed counter can plausibly reach 15-20 percent before the head gate completes because it is derived from selected-file `fileProgress` (`src/core/stream/StreamEngine.cpp:193`, `src/ui/pages/stream/StreamPlayerController.cpp:125`), not progress toward the startup gate (`src/core/stream/StreamEngine.cpp:206`). References frame startup around serving the requested range/read stream or streaming-server URL, not a visible total-file percent gate: Peerflix pumps the requested range stream (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L150`), and Stremio converts magnets to streaming-server URLs (`C:/Users/Suprabha/Downloads/Stremio Reference/stremio-video-master/stremio-video-master/src/withStreamingServer/createTorrent.js:10`).
- P1 - The local HTTP server is not the hot path and would still allow sparse reads after timeout. Our `waitForPieces()` returns false after 15 seconds (`src/core/stream/StreamHttpServer.cpp:74`, `src/core/stream/StreamHttpServer.cpp:85`), but the serving loop ignores that return value and reads the file anyway (`src/core/stream/StreamHttpServer.cpp:207`, `src/core/stream/StreamHttpServer.cpp:210`). Peerflix and Webtor instead place the HTTP response body on torrent-aware readers (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L168`, `https://github.com/webtor-io/torrent-web-seeder/blob/a2eca2148c1ec24dba9cf4569391906bb0182d2d/server/services/web_seeder.go#L221`).
- P1 - Resume/seek into an incomplete torrent is not pre-gated at the target offset. `StreamPage` passes a saved `streamResumeSec` to `VideoPlayer` (`src/ui/pages/StreamPage.cpp:1267`, `src/ui/pages/StreamPage.cpp:1277`), and the sidecar issues `av_seek_frame()` immediately (`native_sidecar/src/video_decoder.cpp:322`). libtorrent's deadline API is designed to retarget priorities when the playback point changes (`https://libtorrent.org/reference-Torrent_Handle.html`), and Webtor restarts/injects seek parameters through its session/transcode flow (`https://github.com/webtor-io/content-transcoder/blob/90b9c9f2a700a4a78325137b815bab7fc45532f2/services/transcode_run.go#L117`). A secondary observed issue is that if the sidecar is already running, `VideoPlayer` sends `sendOpen(filePath)` without `m_pendingStartSec` (`src/ui/player/VideoPlayer.cpp:325`), so resume behavior depends on sidecar state.
- P1 - The sidecar already has HTTP stall buffering/retry behavior, but normal magnet playback bypasses the HTTP path so that behavior cannot protect torrent playback. HTTP paths get reconnect and a 30-second retry loop (`native_sidecar/src/video_decoder.cpp:166`, `native_sidecar/src/video_decoder.cpp:834`), while local paths emit `DECODE_FAILED` on read errors (`native_sidecar/src/video_decoder.cpp:850`). References keep torrent playback behind HTTP/read-stream layers, so decoder stalls are expressed as blocked reads/range responses rather than plain-file sparse reads.

The header says the magnet path returns a local HTTP URL (`src/core/stream/StreamEngine.h:54`), while the implementation explicitly returns a direct file path (`src/core/stream/StreamEngine.cpp:219`). This does not by itself explain the user symptom, but it hides which code path is expected to enforce torrent-frontier blocking.

This is lower priority than the sparse-file and priority-window issues.

## Hypothesized root causes (Agent 4 to validate)

Hypothesis — The random forward jumps happen when ffmpeg reads past the downloaded frontier from a sparse local file; the demuxer encounters zero-filled or incomplete regions and resynchronizes to a later valid timestamp instead of blocking behind torrent piece availability. Agent 4 to validate

Hypothesis — The 15-20 percent startup buffer symptom is the visible result of showing whole selected-file progress while the real gate is first-2-MB contiguity, combined with sequential-download behavior that can let total file progress advance while one early piece remains missing. Agent 4 to validate

Hypothesis — `sequential_download` plus selected-file priority is not strong enough for media streaming because it does not continuously retarget the current playback window, resume target, or ffmpeg read-ahead range the way libtorrent deadlines or torrent-stream critical pieces do. Agent 4 to validate

Hypothesis — The local HTTP server was intended as the torrent-aware read boundary, but the current direct-file return path bypasses it; even if re-enabled, the ignored `waitForPieces()` timeout would still permit sparse reads under slow-piece conditions. Agent 4 to validate

Hypothesis — Resume reliability is state-dependent because a running sidecar drops the caller-supplied `startPositionSec`, while a newly-started sidecar receives it; target-offset prebuffering is absent in both cases. Agent 4 to validate

Hypothesis — `flush_cache()` reduces one class of stale sparse reads at startup but does not create an ongoing blocking contract for later ffmpeg reads, so it cannot prevent mid-stream decoder reads from outrunning libtorrent. Agent 4 to validate
