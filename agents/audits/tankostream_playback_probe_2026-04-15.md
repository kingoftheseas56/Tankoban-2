# Audit — tankostream_playback_probe — 2026-04-15
By Agent 7 (Codex). For Agent 4 (Stream & Sources).

Reference comparison: FFmpeg libavformat HTTP/TCP, Peerflix, mpv demux_lavf, QMPlay2 FFmpeg demuxer.

Scope: This audit covers the cold-start torrent playback path from StreamEngine URL creation through StreamPage, VideoPlayer, SidecarProcess, native_sidecar command handling, sidecar FFmpeg probe/open/read surfaces, and StreamHttpServer response semantics for FFmpeg probes. It compares our current HTTP-backed torrent handoff against Peerflix raw HTTP serving, mpv/QMPlay2 AVIO/interrupt patterns, and FFmpeg's default HTTP behavior. Out of scope: UI parity work, Phase 4+ search work, detail view work, addon management, subtitle rendering quality, audio/codec feature work, and any `src/` or sidecar code changes.

### HTTP server behavior at probe time

StreamHttpServer creates a new `QTcpSocket` around the accepted socket descriptor in a worker thread. If `setSocketDescriptor()` fails it returns; if `waitForReadyRead(5000)` fails it returns without writing any HTTP response (`src/core/stream/StreamHttpServer.cpp:114`, `src/core/stream/StreamHttpServer.cpp:116`, `src/core/stream/StreamHttpServer.cpp:117`, `src/core/stream/StreamHttpServer.cpp:128`, `src/core/stream/StreamHttpServer.cpp:129`).

The request parser reads the request line, then reads only currently available complete header lines while `socket.canReadLine()` is true (`src/core/stream/StreamHttpServer.cpp:132`, `src/core/stream/StreamHttpServer.cpp:133`, `src/core/stream/StreamHttpServer.cpp:135`). It supports `GET` and `HEAD`, validates `/stream/{hash}/{fileIndex}`, parses a single `Range` header, and returns either `200 OK`, `206 Partial Content`, or `416 Range Not Satisfiable` with `Content-Length`, `Accept-Ranges: bytes`, `Cache-Control: no-store`, and `Connection: close` (`src/core/stream/StreamHttpServer.cpp:179`, `src/core/stream/StreamHttpServer.cpp:185`, `src/core/stream/StreamHttpServer.cpp:195`, `src/core/stream/StreamHttpServer.cpp:209`, `src/core/stream/StreamHttpServer.cpp:212`, `src/core/stream/StreamHttpServer.cpp:217`, `src/core/stream/StreamHttpServer.cpp:219`, `src/core/stream/StreamHttpServer.cpp:220`, `src/core/stream/StreamHttpServer.cpp:221`, `src/core/stream/StreamHttpServer.cpp:223`).

For `HEAD`, StreamHttpServer writes headers and closes (`src/core/stream/StreamHttpServer.cpp:226`, `src/core/stream/StreamHttpServer.cpp:227`, `src/core/stream/StreamHttpServer.cpp:229`). For `GET`, it writes the response header before opening/seeking the file (`src/core/stream/StreamHttpServer.cpp:233`, `src/core/stream/StreamHttpServer.cpp:238`, `src/core/stream/StreamHttpServer.cpp:239`, `src/core/stream/StreamHttpServer.cpp:240`).

## Reference behavior

### FFmpeg libavformat HTTP/TCP

FFmpeg's HTTP protocol supports options for `seekable`, custom headers, `user_agent`, `method`, `reconnect`, `reconnect_streamed`, and retry limits (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L169-L206`). It parses `Content-Length`, `Content-Range`, and `Accept-Ranges: bytes`; `Accept-Ranges: bytes` marks the HTTP resource as seekable when FFmpeg is auto-detecting seekability (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1271-L1280`).

FFmpeg's HTTP request builder defaults to `GET`, not `HEAD`, unless the caller overrides `method`. It sends default `User-Agent`, `Accept: */*`, `Connection: close`, `Host`, and deliberately sends a `Range` header even during probing when seekability/range conditions require it (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1537-L1550`, `https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1573-L1614`). Based on this source, FFmpeg probe failures here should be investigated as GET/range response failures first; HEAD support matters but is not the likely first request surface.

The exact FFmpeg log string in our sidecar appears in `http_read_header()`: if `http_get_line(...)` fails while reading the response status/header lines, FFmpeg logs `Error reading HTTP response: ...` and returns that error (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1434-L1452`). This is evidence that the failure is before FFmpeg has accepted a complete HTTP response header, not a later media packet decode failure.

FFmpeg TCP defaults to a 5 second open timeout, and when `rw_timeout` is set it applies that to socket I/O waiting (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/tcp.c#L158-L176`, `https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/tcp.c#L269-L279`). Our `probe_file()` sets `timeout=60000000` and `rw_timeout=30000000`, so the captured 5 second failure is not explained by those sidecar options alone.

### Peerflix

Peerflix uses raw HTTP URLs backed by torrent file streams. It parses the request's `Range` header, always sets `Accept-Ranges: bytes` and `Content-Type`, sets `Content-Length`, handles `HEAD` by ending after headers, and pumps `file.createReadStream()` or `file.createReadStream(range)` into the HTTP response (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L150-L168`). It also sets socket timeouts to a very long interval for playback connections (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L171-L172`).

Peerflix therefore supports the same broad architecture as our Option A: give the player an HTTP URL, not a sparse file path or FIFO. The key reference difference is that Peerflix's request path writes HTTP headers in the normal Node response flow and then uses a torrent-backed stream that blocks/delivers data under the response rather than exposing sparse bytes directly (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L157-L168`).

### mpv

mpv's lavf demuxer has explicit read and seek callbacks (`mp_read`, `mp_seek`, `mp_read_seek`) that route libavformat reads through mpv's own stream abstraction (`https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L292-L349`). For normal demuxers with an owned stream, mpv allocates an `AVIOContext`, attaches it to `avfc->pb`, sets `read_seek` and `seekable`, and then calls `avformat_open_input()` (`https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1033-L1050`, `https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1106-L1108`).

mpv also installs an interrupt callback and custom `io_open`/`io_close` behavior around libavformat (`https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1070-L1083`). This is the mature-player pattern for controlling read, seek, cancellation, nested references, and timeout semantics rather than relying only on libavformat's URLProtocol layer.

### QMPlay2

QMPlay2's FFmpeg demuxer installs an interrupt callback on `AVFormatContext` before opening (`https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L782-L788`). On Android/local-file paths it uses a custom `AVIOContext` over `QFile`, and for Ogg helper paths it can attach a custom `pb` before opening (`https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L790-L805`).

QMPlay2 opens FFmpeg in an `OpenFmtCtxThr` helper and then calls `avformat_find_stream_info()` only after that open succeeds (`https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L814-L818`, `https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L857-L858`). Its read loop treats invalid data / exit errors as recoverable up to a large threshold before declaring an error (`https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L557-L599`).

## Gaps (ranked P0 / P1 / P2)

### P0 — FFmpeg is failing before it receives a complete HTTP response header

The current captured failure is no longer consistent with URL mangling or a missing `probe_file()` HTTP-options rebuild. `_player_debug.txt` shows the exact lowercase HTTP URL in the sidecar open payload, and `sidecar_debug_live.log` shows `probe_file: HTTP mode enabled` for that same URL (`_player_debug.txt:34889`, `sidecar_debug_live.log:22071`). FFmpeg then logs `Error reading HTTP response: End of file`, which its own source emits while reading HTTP response headers (`sidecar_debug_live.log:22072`, `https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1448-L1451`).

The local HTTP server still has an early path that returns without sending any HTTP status line if the worker socket sees no readable request bytes within 5 seconds (`src/core/stream/StreamHttpServer.cpp:128`, `src/core/stream/StreamHttpServer.cpp:129`). The captured open-to-error interval is approximately the same 5 seconds (`_player_debug.txt:34889`, `_player_debug.txt:34895`). That makes this a P0 diagnostic gap: we do not yet have server-side logs proving whether the FFmpeg GET request reached `handleConnection()`, whether the request line was read, or whether a `200/206` header was written.

Reference impact: Peerflix's request handler writes normal HTTP headers for both full and range requests before pumping torrent data (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L153-L168`). FFmpeg's HTTP layer expects a valid status line/header set before any media probing can proceed (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1434-L1452`).

### P0 — Accepted socket ownership is not demonstrated safe

StreamHttpServer detaches the `QTcpSocket*` returned by `nextPendingConnection()`, deliberately keeps it alive, and constructs a second `QTcpSocket` from the same descriptor inside `QtConcurrent::run()` (`src/core/stream/StreamHttpServer.cpp:452`, `src/core/stream/StreamHttpServer.cpp:453`, `src/core/stream/StreamHttpServer.cpp:454`, `src/core/stream/StreamHttpServer.cpp:455`, `src/core/stream/StreamHttpServer.cpp:456`, `src/core/stream/StreamHttpServer.cpp:457`, `src/core/stream/StreamHttpServer.cpp:458`, `src/core/stream/StreamHttpServer.cpp:462`, `src/core/stream/StreamHttpServer.cpp:463`). The code comments acknowledge that the original socket is not deleted because that would close the descriptor (`src/core/stream/StreamHttpServer.cpp:454`, `src/core/stream/StreamHttpServer.cpp:459`).

The audit cannot assert this is the root cause, but the implementation has no local evidence that only the worker socket owns readiness notifications and reads for that descriptor. The observed 5 second EOF-on-header failure is exactly the kind of user-visible failure that would occur if the worker socket never receives the request bytes and returns through the no-response path (`src/core/stream/StreamHttpServer.cpp:128`, `src/core/stream/StreamHttpServer.cpp:129`).

Reference impact: Peerflix keeps request handling inside the HTTP server's native request/response object and does not re-wrap an accepted descriptor in a second socket object (`https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L150-L168`). mpv avoids this class at the demux boundary by routing libavformat through controlled read/seek callbacks when it owns the stream (`https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1039-L1050`).

### P1 — Probe is a single-shot gate; decoder HTTP retry logic cannot help it

The sidecar's open worker calls `probe_file()` before constructing/starting AudioDecoder and VideoDecoder. If `avformat_open_input()` or `avformat_find_stream_info()` fails in `probe_file()`, open exits immediately with `OPEN_FAILED` (`native_sidecar/src/main.cpp:270`, `native_sidecar/src/main.cpp:272`, `native_sidecar/src/main.cpp:273`, `native_sidecar/src/main.cpp:274`). The VideoDecoder and AudioDecoder stall retry loops are therefore unreachable on initial probe failure (`native_sidecar/src/video_decoder.cpp:863`, `native_sidecar/src/video_decoder.cpp:872`, `native_sidecar/src/audio_decoder.cpp:414`, `native_sidecar/src/audio_decoder.cpp:417`).

The probe HTTP-options hotfix is present and active, but the captured failure occurs at the response-header layer, before the decoder loops exist (`native_sidecar/src/demuxer.cpp:40`, `native_sidecar/src/demuxer.cpp:48`, `sidecar_debug_live.log:22071`, `sidecar_debug_live.log:22072`). mpv/QMPlay2 both put more of the demux open behind controlled stream/interruption machinery, while our probe path is currently a direct `avformat_open_input()` against FFmpeg's HTTP URLProtocol (`https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1039-L1050`, `https://github.com/zaps166/QMPlay2/blob/e7784aafdcc26c7fe4d9ea249c5bd156a62f7a8b/src/modules/FFmpeg/FormatContext.cpp#L782-L818`).

### P1 — Subtitle preload is still an unaudited HTTP-hostile avformat surface

This does not explain the initial `Cannot open file: probe failed` error, because it is invoked only after successful probe/open and on subtitle switching. It is still an HTTP gap in the same playback stack: `preload_subtitle_packets()` opens `g_current_path` with no HTTP options and then reads packets until EOF (`native_sidecar/src/main.cpp:762`, `native_sidecar/src/main.cpp:766`, `native_sidecar/src/main.cpp:789`, `native_sidecar/src/main.cpp:857`, `native_sidecar/src/main.cpp:868`). On a torrent-backed HTTP URL, that can create a later failure mode or long blocking read independent of the fixed probe/video/audio open paths.

### P1 — Piece deadline behavior is called but not observable

The current code calls `set_piece_deadline()` for the head window and uses the 5 MB contiguous gate before launching (`src/core/stream/StreamEngine.cpp:559`, `src/core/stream/StreamEngine.cpp:576`, `src/core/stream/StreamEngine.cpp:262`). TorrentEngine does not log how many deadline calls were accepted, whether the torrent handle had metadata, or what the contiguous frontier looked like during FFmpeg probe reads (`src/core/torrent/TorrentEngine.cpp:776`, `src/core/torrent/TorrentEngine.cpp:783`, `src/core/torrent/TorrentEngine.cpp:787`, `src/core/torrent/TorrentEngine.cpp:792`).

This is lower probability for the captured EOF-before-header failure, but it remains relevant if server tracing later proves headers are sent and FFmpeg fails during `avformat_find_stream_info()` or later range reads. FFmpeg may issue ranged GETs during probing (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1589-L1605`), while our hard startup gate guarantees only the first 5 MB and `probe_file()` allows a 20 MB probe budget (`src/core/stream/StreamEngine.cpp:201`, `native_sidecar/src/demuxer.cpp:46`).

### P2 — HTTP scheme detection is case-sensitive inside the sidecar

SidecarProcess preserves HTTP URLs case-insensitively (`src/ui/player/SidecarProcess.cpp:143`), and StreamEngine generates lowercase `http://` (`src/core/stream/StreamEngine.cpp:670`), so this is not implicated in the captured StreamEngine URL path. The sidecar's `probe_file()`, VideoDecoder, and AudioDecoder check only lowercase prefixes (`native_sidecar/src/demuxer.cpp:38`, `native_sidecar/src/video_decoder.cpp:167`, `native_sidecar/src/audio_decoder.cpp:151`). A direct URL source with uppercase `HTTP://` would skip the HTTP options blocks.

### P2 — HEAD handling appears acceptable, but is probably not the first failing probe path

StreamHttpServer implements `HEAD` with `Content-Length`, `Accept-Ranges: bytes`, and `Connection: close` (`src/core/stream/StreamHttpServer.cpp:219`, `src/core/stream/StreamHttpServer.cpp:220`, `src/core/stream/StreamHttpServer.cpp:221`, `src/core/stream/StreamHttpServer.cpp:226`, `src/core/stream/StreamHttpServer.cpp:227`). FFmpeg's HTTP source defaults to `GET`, not `HEAD`, and deliberately emits Range headers during probe/seekability discovery (`https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1547-L1550`, `https://github.com/FFmpeg/FFmpeg/blob/3f9e04b48958ca9cd75ce9f36feb30cd6d861add/libavformat/http.c#L1589-L1605`). HEAD conformance should stay covered, but the current evidence points to GET response-header failure.

### P2 — VideoPlayer builds a directory playlist from an HTTP URL

When no playlist is passed, VideoPlayer builds a local `QDir(QFileInfo(filePath).absolutePath())` playlist even when `filePath` is an HTTP URL (`src/ui/player/VideoPlayer.cpp:283`, `src/ui/player/VideoPlayer.cpp:285`). This does not alter the sent sidecar path (`src/ui/player/VideoPlayer.cpp:293`, `src/ui/player/VideoPlayer.cpp:342`, `src/ui/player/VideoPlayer.cpp:412`), and there is no evidence it contributes to the probe failure. It is only a small side-effect to keep in mind while tracing playback launch.
