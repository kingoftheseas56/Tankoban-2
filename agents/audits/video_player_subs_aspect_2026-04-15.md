# Audit — video_player_subs_aspect — 2026-04-15

By Agent 7 (Codex). For Agent 3 (Video Player).
Reference comparison: mpv, IINA, QMPlay2, libass.
Scope: Audited Tankoban's subtitle rendering path from main-app track selection through sidecar libass/PGS rendering and D3D/SHM presentation, plus aspect-ratio selection and fullscreen viewport math. Out of scope: audio, stream buffering, codec preference, UI styling, and non-player library pages.

## Reference behavior

- libass documents that the renderer should be configured with storage size, frame size, and fonts before rendering; frame size is required before using a renderer, storage size is required for correct results, and pixel aspect may be needed when frame/display/storage aspect diverge ([libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L372), [libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L399), [libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L418), [libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L471)). `ass_render_frame` returns an `ASS_Image` list for a timestamp, and `ass_process_chunk` adds packet events by timecode/duration ([libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L617), [libass ass.h](https://github.com/libass/libass/blob/master/libass/ass.h#L730)).
- mpv initializes ASS tracks with codec private data, configures the ASS renderer for the current OSD/video resolution, applies pixel aspect/storage size, renders with `ass_render_frame`, and packs the returned ASS images into subtitle bitmaps (`https://github.com/mpv-player/mpv/blob/master/sub/sd_ass.c#L277`, `https://github.com/mpv-player/mpv/blob/master/sub/sd_ass.c#L763`, `https://github.com/mpv-player/mpv/blob/master/sub/sd_ass.c#L767`, `https://github.com/mpv-player/mpv/blob/master/sub/sd_ass.c#L780`).
- mpv's GPU OSD path treats libass results as GPU-drawn sub-bitmaps: it prepares shader code for `SUBBITMAP_LIBASS`, generates quads for each sub-bitmap, enables blend factors, and dispatches the draw into the same render target flow (`https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L222`, `https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L267`, `https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L301`, `https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L304`).
- mpv aspect calculation uses display size from video image parameters, monitor pixel aspect, margins, crop validity, and rotation before producing source/destination rectangles and OSD margins (`https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L128`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L139`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L155`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L158`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L172`).
- IINA delegates playback/subtitle/aspect behavior to libmpv: it observes `sid`, `sub-visibility`, and related subtitle properties, sets `keepaspect` to yes, toggles mpv fullscreen through the mpv property, and constrains its video view to fill the content view (`https://github.com/iina/iina/blob/develop/iina/MPVController.swift#L126`, `https://github.com/iina/iina/blob/develop/iina/MPVController.swift#L145`, `https://github.com/iina/iina/blob/develop/iina/MPVController.swift#L674`, `https://github.com/iina/iina/blob/develop/iina/MainWindowController.swift#L275`, `https://github.com/iina/iina/blob/develop/iina/MainWindowController.swift#L727`).
- QMPlay2 computes video size with `QSizeF(aspect, 1.0).scaled(..., Qt::KeepAspectRatio)`, computes a subtitle rectangle from the same aspect-scaled space, and renders OSD after rendering video in its Vulkan path (`https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/VideoOutputCommon.cpp#L98`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/VideoOutputCommon.cpp#L111`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L654`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L656`). Its OSD renderer builds OSD memory/pipelines and positions subtitle images inside the computed subtitle rectangle (`https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L778`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L863`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L885`).

## Gaps (ranked P0 / P1 / P2)

**P0 (user-blocking or severely degrading):**
- Subtitle composition target can diverge from the displayed target when zero-copy is active. Observed: the sidecar copies the decoded hardware frame to the shared D3D texture before subtitle blending, disables the zero-copy short-circuit when subtitles are active, but still skips `present_cpu()` when `d3d_gpu_copied` is true; the main app displays the imported D3D texture before considering SHM (`native_sidecar/src/video_decoder.cpp:407`, `native_sidecar/src/video_decoder.cpp:431`, `native_sidecar/src/video_decoder.cpp:730`, `src/ui/player/FrameCanvas.cpp:606`). Reference: mpv and QMPlay2 draw subtitle/OSD bitmaps into the same GPU output sequence as video (`https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L301`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L654`). Impact: the active display path may show an un-subtitled texture while the blended subtitle frame exists only in SHM.
- Runtime trace may not be executing the audited sidecar source. Observed: current resolver can fall back to the old GroundWork executable, and the supplied log shows that path plus stale-command errors (`src/ui/player/SidecarProcess.cpp:58`, `_player_debug.txt:33526`, `_player_debug.txt:33552`). Reference: IINA delegates to one libmpv controller/runtime for property state rather than carrying a transitional external-binary fallback in the playback path (`https://github.com/iina/iina/blob/develop/iina/MPVController.swift#L840`, `https://github.com/iina/iina/blob/develop/iina/MPVController.swift#L864`). Impact: observations from `_player_debug.txt` may reflect a binary that lacks current sidecar changes, including subtitle fixes.

**P1 (user-visible shortfall):**
- The main-app QLabel subtitle overlay is not fed by the current native sidecar renderer. Observed: `SubtitleOverlay` only receives `subtitle_text`; current sidecar subtitle source does frame blending and has no `subtitle_text` emitter (`src/ui/player/VideoPlayer.cpp:234`, `native_sidecar/src/main.cpp:489`, `native_sidecar/src/subtitle_renderer.cpp:422`). Reference: mpv/QMPlay2 handle libass output as image/sub-bitmap composition, not as a separate text label (`https://github.com/mpv-player/mpv/blob/master/sub/sd_ass.c#L781`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L798`). Impact: any debugging that expects QLabel text to prove subtitle rendering will stay silent even if the frame-blend path is active.
- Same-track `set_tracks` does not reload/preload the renderer. Observed: `handle_set_tracks` reloads only when `new_sub_id != g_active_sub_id`, while startup sets `g_active_sub_id` to the first subtitle stream before the open-time preload (`native_sidecar/src/main.cpp:585`, `native_sidecar/src/main.cpp:614`, `native_sidecar/src/main.cpp:854`). Reference: libass packet/event state is explicitly track-based, with codec-private and packet chunks feeding the track used by `ass_render_frame` (`https://github.com/libass/libass/blob/master/libass/ass.h#L714`, `https://github.com/libass/libass/blob/master/libass/ass.h#L730`, `https://github.com/libass/libass/blob/master/libass/ass.h#L617`). Impact: re-selecting an already-active subtitle stream has no restorative load/preload path if the track state is stale or incomplete.
- Natural aspect uses storage frame dimensions only. Observed: `FrameCanvas` computes `frameAspect` from `m_frameW / m_frameH`; sidecar probing captures only `codecpar->width/height`, and no SAR/DAR/crop/rotation travels to the viewport (`src/ui/player/FrameCanvas.cpp:633`, `native_sidecar/src/demuxer.cpp:52`). Reference: mpv derives display size from image params and accounts for crop/rotation/monitor PAR; libass also documents storage size and pixel aspect as distinct concepts (`https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L139`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L155`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L158`, `https://github.com/libass/libass/blob/master/libass/ass.h#L471`). Impact: sources with non-square pixels, display matrices, or crop metadata can render at the wrong natural aspect until a user forces a fixed menu ratio.

**P2 (polish):**
- The QLabel overlay's z-order over a native D3D HWND is unproven from code. Observed: `FrameCanvas` is `WA_PaintOnScreen`/`WA_NativeWindow`, while `SubtitleOverlay` is a sibling raised in Qt widget order (`src/ui/player/FrameCanvas.cpp:23`, `src/ui/player/VideoPlayer.cpp:2036`, `src/ui/player/VideoPlayer.cpp:2038`). Reference: mpv and QMPlay2 avoid sibling-widget subtitle composition by drawing OSD/subtitle images in the renderer path (`https://github.com/mpv-player/mpv/blob/master/video/out/gpu/osd.c#L304`, `https://github.com/zaps166/QMPlay2/blob/master/src/qmplay2/vulkan/VulkanWindow.cpp#L768`). Impact: if a future text-event overlay path is revived, it may still fail visually on Windows native-window stacking.
- `m_frameW/m_frameH` lifetime is not explicitly tied to the current media lifecycle. Observed: `detachD3D11Texture()` releases D3D resources and clears `m_d3dActive`, but does not clear frame dimensions; `tearDownD3D()` also leaves frame dimensions untouched (`src/ui/player/FrameCanvas.cpp:1238`, `src/ui/player/FrameCanvas.cpp:1245`, `src/ui/player/FrameCanvas.cpp:361`). Reference: mpv recomputes source/destination rectangles from current video params and window dimensions in the aspect helper (`https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L128`, `https://github.com/mpv-player/mpv/blob/master/video/out/aspect.c#L135`). Impact: transient or stale aspect decisions are harder to rule out during fullscreen/device transitions.

## Hypothesized root causes (Agent 3 to validate)

- **Hypothesis —** Subtitles are being blended into the CPU/SHM frame, but zero-copy display continues to show the pre-blend shared D3D texture because `d3d_gpu_copied` skips the later CPU presenter upload while `FrameCanvas` prefers `m_importedSrv`. **Agent 3 to validate.**
- **Hypothesis —** The `SubtitleOverlay` QLabel is silent by design in the current native path because the sidecar renderer no longer emits `subtitle_text`; the active subtitle path is image blending, not text emission. **Agent 3 to validate.**
- **Hypothesis —** Re-selecting the currently active subtitle stream can fail to repair an incomplete renderer state because `handle_set_tracks` skips clear/load/preload when the requested subtitle id equals `g_active_sub_id`. **Agent 3 to validate.**
- **Hypothesis —** At least some aspect complaints are display-aspect metadata issues rather than fullscreen-only timing issues: the viewport uses coded frame dimensions but does not ingest SAR/DAR/crop/rotation metadata. **Agent 3 to validate.**
- **Hypothesis —** Some log evidence came from a stale GroundWork sidecar binary, which may explain command support and behavior mismatches against `native_sidecar/src`. **Agent 3 to validate.**
- **Hypothesis —** If the aspect error appears only during fullscreen transitions and self-corrects after later frames, stale `m_frameW/m_frameH` or delayed D3D import around `resizeEvent` may be involved; if it persists for the whole file, the storage-vs-display-aspect path is more likely. **Agent 3 to validate.**
