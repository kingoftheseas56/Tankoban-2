### 2. Readest Implementation Pattern

Observations:
- Readest implements Edge TTS through a direct Edge Read Aloud client, not Web Speech. `EdgeTTSClient.ts` imports `EdgeSpeechTTS` from `@/libs/edgeTTS` (`EdgeTTSClient.ts:1-4`), and `edgeTTS.ts` declares the endpoint `wss://speech.platform.bing.com/consumer/speech/synthesize/readaloud/edge/v1` plus the trusted client token and Chromium/Edge version strings (`edgeTTS.ts:32-36`).
- Readest ships its own static Edge voice table. `edgeTTS.ts:38-113` contains locale-to-neural-voice mappings including `en-IN-NeerjaExpressiveNeural`, `en-IN-NeerjaNeural`, `en-IN-PrabhatNeural`, and many `en-US` voices. The generated list becomes `EdgeSpeechTTS.voices` at `edgeTTS.ts:286-288`.
- Readest probes Edge availability at startup by creating a short synthesis request using `en-US-AriaNeural` (`EdgeTTSClient.ts:35-47`). If WSS init fails, it retries an authenticated HTTPS route when possible or dispatches `tts-need-auth` (`EdgeTTSClient.ts:47-56`).
- Readest's WSS request builds `ConnectionId`, `TrustedClientToken`, `Sec-MS-GEC`, and `Sec-MS-GEC-Version` query params (`edgeTTS.ts:325-333`), sets Edge-like headers including user agent, `Origin: chrome-extension://jdiccldimpdaibmpdkjnbmckianbfold`, and random `muid` cookie (`edgeTTS.ts:334-347`), and sends `speech.config` plus SSML messages (`edgeTTS.ts:348-403`).
- Readest requests `wordBoundaryEnabled: true` in Edge protocol metadata config (`edgeTTS.ts:359-368`), but its `EdgeTTSClient` playback UX uses parsed Foliate/SSML marks as sentence chunks. It dispatches a boundary at chunk start (`EdgeTTSClient.ts:127-146`) and reports supported granularities as `['sentence']` (`EdgeTTSClient.ts:312-314`).
- Readest has three transport branches: Tauri WebSocket plugin with headers (`edgeTTS.ts:405-446`), Cloudflare Workers fetch-upgrade with binary-drain sequencing (`edgeTTS.ts:447-583`), and Node/isomorphic-ws with custom headers (`edgeTTS.ts:583-630`). Its package includes `@tauri-apps/plugin-websocket` and `isomorphic-ws` (`package.json:116`, `package.json:144`).
- Readest's Tauri CSP explicitly allows `wss://speech.platform.bing.com`, `https://edge.microsoft.com`, and broad HTTP/HTTPS connect sources (`src-tauri/tauri.conf.json:16-18`).
- Readest caches generated audio. `EdgeSpeechTTS` has LRU caches for 200 blobs/URLs (`edgeTTS.ts:286-293`), and `createAudioUrl()` hashes the payload, reuses cache hits, creates an MP3 Blob, and caches the object URL (`edgeTTS.ts:642-654`).
- Readest uses a single reusable `HTMLAudioElement` for mark playback (`EdgeTTSClient.ts:118-125`), with pause/resume based on `currentTime` and platform-specific fade compensation for Safari/iOS (`EdgeTTSClient.ts:216-247`).
- Readest preloads ahead. For preload mode it generates audio for the first two marks immediately and the rest in the background (`EdgeTTSClient.ts:76-115`); `TTSController.preloadNextSSML()` gathers up to four upcoming SSML chunks and rewinds synchronously to avoid highlight range races (`TTSController.ts:261-287`).
- Readest's controller initializes Edge first, then native, then Web Speech, and selects the first available or preferred client (`TTSController.ts:74-98`). This means Web Speech is a fallback, not the primary Edge TTS path.
- Readest voice selection merges native, Edge, and Web Speech groups (`TTSController.ts:497-503`), and `setVoice()` selects Edge when the requested Edge voice exists and is not disabled (`TTSController.ts:506-529`).
- Readest's TTS start path guards concurrent starts, stops any existing controller before creating another, initializes media/session behavior, initializes controller/view TTS, then starts from saved location, selected range, or current progress (`useTTSControl.ts:423-520`).
- Readest transport controls include previous paragraph, previous sentence, play/pause, next sentence, and next paragraph (`TTSBar.tsx:53-93`). The panel includes rate, voice, timeout, and sticky TTS bar controls (`TTSPanel.tsx:138-155`, `TTSPanel.tsx:183-196`, `TTSPanel.tsx:270-306`, `TTSPanel.tsx:321-386`).
- Readest throttles rate and voice changes; if TTS is playing it stops, applies the change, and restarts (`useTTSControl.ts:596-629`).
- Readest integrates media-session actions for play, pause, stop, seek forward/backward, next track, and previous track (`useTTSControl.ts:682-715`).

Hypotheses:
- Hypothesis - Agent 2 to validate: Readest's perceived reliability is likely less about Web Speech voice availability and more about owning the Edge network path, owning the voice list, probing early, caching chunks, and using sentence-level mark playback instead of depending on browser-native voice enumeration. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: Readest intentionally avoids word-level UX for Edge playback even though its protocol asks for word metadata; this may be a reliability trade-off that Agent 2 should compare against Tankoban's more ambitious word-boundary scheduler. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: A Tankoban implementation that mirrors Readest's mark/chunk model before attempting word-level highlighting may reduce moving parts while preserving a useful reader-grade TTS experience. Agent 2 to validate.

### 3. The `msedge-tts` / Direct Edge TTS Ecosystem

Observations:
- The Python `rany2/edge-tts` project describes itself as using Microsoft Edge's online text-to-speech service without requiring Edge, Windows, or an API key, and supports CLI media/subtitle output plus voice listing (`rany2/edge-tts` README lines 274-304).
- `rany2/edge-tts` exposes voice choice, voice listing, rate, volume, and pitch controls (`README` lines 301-328).
- `rany2/edge-tts` documents a major SSML constraint: custom SSML was removed because Microsoft permits only SSML that Edge itself could generate, effectively a single `voice` tag with one `prosody` tag (`README` lines 320-322).
- `rany2/edge-tts` constants match Readest's protocol: `speech.platform.bing.com/consumer/speech/synthesize/readaloud`, trusted client token, WSS URL with token, voice-list URL, Chromium version, Edge-like headers, and extension origin (`constants.py` lines 331-371).
- `rany2/edge-tts` generates `Sec-MS-GEC` by using corrected current time in Windows file-time form, rounding to five minutes, concatenating with trusted client token, then SHA256 hashing (`drm.py` lines 716-757). It also carries clock-skew correction from server dates (`drm.py` lines 672-712).
- `rany2/edge-tts` sends `speech.config` requesting either sentence or word boundary metadata and MP3 output (`communicate.py` lines 2220-2247), then opens the WSS URL with `ConnectionId`, generated `Sec-MS-GEC`, and Edge headers (`communicate.py` lines 2280-2305).
- `rany2/edge-tts` parses `audio.metadata` frames into `WordBoundary` or `SentenceBoundary` chunks with `offset`, `duration`, and text (`communicate.py` lines 2151-2184).
- `rany2/edge-tts` compensates long-text metadata offsets based on 48 kbps MP3 byte count because the project notes drift/overflow in Microsoft offsets (`communicate.py` lines 2186-2217).
- `rany2/edge-tts` raises `NoAudioReceived` when the service returns no audio (`communicate.py` lines 2449-2455), and issue #440 shows a real reported stack trace with that exception on 2025-11-29 (`issue #440` lines 178-214). Issue #442 shows another December 2025 report of TTS suddenly stopping (`issue #442` lines 178-221).
- The Node/Bun `@andresaya/edge-tts` package advertises the same no-Edge/no-Windows/no-API-key Edge TTS access and TypeScript support, installable via npm or bun (GitHub lines 270-280, 699-701). It acknowledges `rany2/edge-tts` as groundwork (GitHub lines 690-696).
- The `travisvn/edge-tts-client` package is explicitly a TypeScript Edge Read Aloud client for Node/browser, supports audio streaming, and exposes `setMetadata`, `toStream`, and `close` (`travisvn` README lines 250-311).
- Microsoft official Speech docs are not Edge Read Aloud docs, but they document adjacent backend behavior: speech synthesis supports many voices and languages, voice/language mismatch can produce no synthesized audio, word-boundary events include audio offsets and text positions, and event delivery can be faster than playback, so consumers must synchronize themselves (Microsoft speech synthesis lines 85-99, 213-220, 909-915).
- Microsoft SSML docs describe prosody, style, role, bookmark, phoneme, and lexicon features for the supported Azure Speech service (`speech-synthesis-markup-voice` lines 122-134 and 287-289; `speech-synthesis-markup-structure` lines 238-255; `speech-synthesis-markup-pronunciation` lines 61-91 and 99-136). Edge Read Aloud consumer protocol may support only a subset, per `rany2`.

Hypotheses:
- Hypothesis - Agent 2 to validate: The most mature public direct-Edge pattern is not Web Speech; it is the Edge Read Aloud WSS protocol with static/dynamic Edge voice data, `Sec-MS-GEC`, Edge-like headers, MP3 binary frames, and metadata parsing. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: Direct Edge TTS reliability risks are concentrated in protocol drift, clock skew/token generation, network blocks, endpoint changes, voice deprecations, and `NoAudioReceived` cases rather than local browser voice availability. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: Because the Edge consumer protocol is unofficial while Azure Speech is official, Agent 2 should separate "protocol mechanics we can learn from Azure docs" from "product/API support guarantees" when designing the Edge-only implementation. Agent 2 to validate.

### 4. Web Speech API vs Direct WebSocket Trade-off

Observations:
- Web Speech `getVoices()` is user-agent dependent and must return an empty list when voices are unavailable or not yet known; `voiceschanged` is how async voice availability is surfaced (Web Speech spec lines 629-634; MDN getVoices lines 194-247; MDN voiceschanged lines 192-231).
- Web Speech boundary events are conditional. The spec says mark/boundary events must fire if the speech synthesis engine provides them, and `charIndex`, `charLength`, and `elapsedTime` can fall back to zero when unsupported (`Web Speech spec` lines 663-678). MDN marks the boundary event as limited availability (MDN boundary lines 184-194).
- Web Speech voice selection is indirect. If a voice from `getVoices()` is set on the utterance, the UA uses it; otherwise it uses the UA default voice (`Web Speech spec` lines 640-642).
- Web Speech exposes useful error codes such as `network`, `synthesis-unavailable`, `synthesis-failed`, `language-unavailable`, `voice-unavailable`, `text-too-long`, and `not-allowed` (`Web Speech spec` lines 580-593).
- Readest does not rely on Web Speech for Edge voices; it initializes Edge direct first and only uses Web Speech as a fallback group (`TTSController.ts:74-98`).
- Tankoban current code is already closer to direct-Edge than to Web Speech at the JS engine level, but the backend is missing/stubbed. It has probe, voice, synth, synthStream, cache, MediaSource, and boundary scheduler concepts in `tts_engine_edge.js:175-224`, `tts_engine_edge.js:281-345`, `tts_engine_edge.js:695-785`, and `tts_engine_edge.js:913-945`.
- Direct Edge WebSocket gives the app control over voice inventory, request format, cache keys, audio chunks, metadata frames, retries, and diagnostics. Readest demonstrates those controls with `edgeTTS.ts:325-403`, `edgeTTS.ts:405-630`, and `edgeTTS.ts:642-654`.
- Direct Edge WebSocket also increases maintenance burden because the consumer endpoint is not an official public API, token/header behavior has changed historically, and the ecosystem reports `NoAudioReceived` and sudden outage cases (`rany2` issues #440 and #442).

Hypotheses:
- Hypothesis - Agent 2 to validate: If the goal is "reliable and consistent Edge neural voices", Web Speech is structurally weaker because voice inventory, selected voice, and boundary event fidelity remain under the user agent rather than Tankoban. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: Direct WebSocket is structurally stronger for consistency but only if Agent 2 owns explicit failure states, retry/backoff, token refresh/clock skew, voice-list refresh, request chunking, and offline/network messaging. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: A hybrid path could use direct Edge as primary and Web Speech only as an explicitly labeled degraded fallback, but that may conflict with the project direction to use Edge TTS only if fallback voices are not Edge voices. Agent 2 to validate.

### 5. QtWebEngine Context Specifics

Observations:
- Tankoban Book Reader runs in QtWebEngine via `QWebEngineView` (`BookReader.cpp:71-73`) and injects `window.electronAPI` through a `QWebEngineScript` + `QWebChannel` shim (`BookReader.cpp:110-180`).
- The reader HTML is local-file content. Qt-side settings allow local file URLs but explicitly disallow local content from remote URLs (`BookReader.cpp:75-77`).
- Qt documentation says `LocalContentCanAccessRemoteUrls` allows local-origin documents to access remote resources that would otherwise be blocked and is disabled by default (`QWebEngineSettings` lines 112-116).
- Qt documentation says Qt WebEngine supports the JavaScript WebSocket API for `ws://` and `wss://` servers (`Qt WebEngine features` lines 279-282).
- Qt documentation says MP3 support depends on proprietary audio/video codecs being enabled at Qt build/configuration time (`Qt WebEngine features` lines 50-67). Tankoban's Edge path uses `audio/mpeg` in MediaSource and Blob playback (`tts_engine_edge.js:700-706`, `tts_engine_edge.js:1093-1104`).
- No Qt-side audio output device selection, focus policy for TTS playback, or native TTS hooks were found in `BookReader.cpp` / `BookReader.h`. The only TTS-native hooks are the stubbed `booksTtsEdge` bridge functions (`BookReader.cpp:169-176`).
- Browser WebSocket APIs cannot set arbitrary request headers such as `Origin`, `Cookie`, and Edge-like user-agent. Readest avoids this in Tauri by using `@tauri-apps/plugin-websocket` with headers (`edgeTTS.ts:405-410`) and in Node by using `isomorphic-ws` with headers (`edgeTTS.ts:583-587`).

Hypotheses:
- Hypothesis - Agent 2 to validate: A robust QtWebEngine design likely needs the Edge network client outside ordinary renderer JavaScript, either in a Qt/C++ bridge, helper process, or controlled local service, because the Edge consumer protocol needs headers and local-file remote access policy currently blocks renderer-direct remote calls. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: Before any direct Edge backend is chosen, Agent 2 should verify MP3 decode support in the distributed QtWebEngine build because both Readest and Tankoban current paths assume MP3 object URLs or MediaSource MP3. Agent 2 to validate.
- Hypothesis - Agent 2 to validate: If TTS playback is implemented with `HTMLAudioElement`, Agent 2 should verify output-device behavior and focus/background playback in QtWebEngine rather than assuming desktop-browser behavior. Agent 2 to validate.

## Implementation Strategy Options for Agent 2 to Choose Between

Option A - Stay on Web Speech API and harden only if the codebase is intentionally moved back to Web Speech:
- Potential benefits: smallest backend footprint, browser-native queue/pause/resume API, no unofficial Edge protocol maintenance.
- Trade-offs: Edge neural voice availability is user-agent and runtime dependent; `getVoices()` may be empty until async population; boundary events are conditional; voice identity cannot be made as deterministic as a direct Edge voice table.
- Evidence: Web Speech spec says voice availability is UA-dependent and can return empty while not yet known (spec lines 629-634); boundary/mark events depend on engine support (spec lines 663-678).
- Hypothesis - Agent 2 to validate: This option likely cannot deliver "Readest-like" Edge voice consistency unless QtWebEngine reliably exposes the desired Edge voices and boundary events on the target Windows runtime. Agent 2 to validate.

Option B - Direct Edge Read Aloud backend behind Tankoban's existing `booksTtsEdge` bridge:
- Potential benefits: aligns with current `tts_engine_edge.js` bridge contract; avoids renderer header limitations; can own voices, token generation, retries, diagnostics, audio payload shape, and cache; closest to Readest and public `edge-tts` libraries.
- Trade-offs: requires native/helper implementation of unofficial Edge consumer protocol; must track `Sec-MS-GEC` and endpoint changes; must handle `NoAudioReceived`, network blocks, and voice drift; must ensure license/dependency compatibility if code is borrowed.
- Evidence: Readest owns WSS endpoint, token params, headers, transports, cache, and voice table (`edgeTTS.ts:32-36`, `325-403`, `405-654`); `rany2/edge-tts` implements the same protocol and token generation (`constants.py` lines 331-371; `drm.py` lines 716-757).
- Hypothesis - Agent 2 to validate: This option is the most compatible with current Tankoban JS structure because `tts_engine_edge.js` already expects `probe`, `getVoices`, `synth`, `synthStream`, `cancelStream`, `warmup`, and `resetInstance`. Agent 2 to validate.

Option C - Direct Edge in renderer JavaScript:
- Potential benefits: fewer native bridge methods, closer to Readest's TS code shape, faster iteration in reader JS.
- Trade-offs: current local-origin remote access is disabled; browser WebSocket cannot set arbitrary Edge headers; Qt CSP/settings and file-origin behavior must be changed carefully; the implementation may fail for protocol reasons outside app logic.
- Evidence: `BookReader.cpp:75-77` disables local content remote access; Qt docs describe that setting as blocking remote resources from local origins (Qt `QWebEngineSettings` lines 112-116); Readest uses Tauri plugin/Node paths to set WebSocket headers (`edgeTTS.ts:405-410`, `583-587`).
- Hypothesis - Agent 2 to validate: Renderer-only WSS is unlikely to match Readest reliability in QtWebEngine without a companion native transport because the Edge consumer protocol depends on non-browser WebSocket control. Agent 2 to validate.

Option D - Staged hybrid inside the Edge-only constraint:
- Potential benefits: bridge/direct Edge primary for synthesis; sentence-level playback and highlight first; word-boundary streaming later; Web Speech used only for capability diagnostics or not at all depending on Agent 2/Agent 0 interpretation of "Edge only".
- Trade-offs: requires explicit product states to avoid misleading "available" checks; sentence-first may be less visually granular than current Tankoban word-highlight ambition; if Web Speech fallback voices are not Edge voices, this may violate standing direction.
- Evidence: Readest Edge client supports sentence granularity (`EdgeTTSClient.ts:312-314`) while public Edge protocol can provide WordBoundary metadata (`communicate.py` lines 2151-2184).
- Hypothesis - Agent 2 to validate: A staged Edge-only hybrid that keeps direct Edge as the only synthesizer but delays word-level highlighting may produce a more reliable first shipped TTS batch. Agent 2 to validate.
