### Teardown paths observed

- User closes player/back/Escape/context back/media stop: `VideoPlayer` emits `closeRequested` from multiple UI paths (`src/ui/player/VideoPlayer.cpp:953`, `src/ui/player/VideoPlayer.cpp:2340`, `src/ui/player/VideoPlayer.cpp:2717`, `src/ui/player/VideoPlayer.cpp:2737`). MainWindow persistently connects that signal to `closeVideoPlayer()` (`src/ui/MainWindow.cpp:127`), and StreamPage adds/removes a per-stream connection in `onReadyToPlay()` (`src/ui/pages/StreamPage.cpp:1706`).
- MainWindow close stops playback immediately (`src/ui/MainWindow.cpp:457`). This is outside StreamPage's session cleanup path, so stream controller/torrent teardown depends on whether StreamPage's close connection also runs before shutdown.
- Buffer-cancel while still buffering calls only `m_playerController->stopStream()` (`src/ui/pages/StreamPage.cpp:221`).
- Source-switch mid-play calls `StreamPage::onSourceActivated()`, which sets the new `_currentEpKey` before calling `StreamPlayerController::startStream()` (`src/ui/pages/StreamPage.cpp:1587`, `src/ui/pages/StreamPage.cpp:1610`). `startStream()` immediately calls `stopStream()` for the old active session (`src/ui/pages/stream/StreamPlayerController.cpp:26`), and that emits `streamStopped()` synchronously (`src/ui/pages/stream/StreamPlayerController.cpp:81`). `StreamPage::onStreamStopped()` then clears `_currentEpKey`, disconnects the player, resets persistence, hides the buffer overlay, and usually calls `showBrowse()` (`src/ui/pages/StreamPage.cpp:1913`). This is a high-risk reentrancy path because old-session teardown runs inside new-session setup.
- `streamFailed` path clears `_currentEpKey`, auto-launch, next-episode state, seek retry, player dynamic connections, and persistence mode, then schedules a 3s delayed `showBrowse()` guarded only by controller inactivity (`src/ui/pages/StreamPage.cpp:1885`).
- Next-episode auto-advance is a special stop-then-start path. The close lambda can show overlay before `stopStream()` (`src/ui/pages/StreamPage.cpp:1717`), `onStreamStopped()` returns while overlay is visible (`src/ui/pages/StreamPage.cpp:1929`), and Play Now later repopulates `m_pendingPlay` and calls `onSourceActivated()` (`src/ui/pages/StreamPage.cpp:1435`).
- Shift+N currently routes through `VideoPlayer::streamNextEpisodeRequested` to `StreamPage::onStreamNextEpisodeShortcut()` with `Qt::UniqueConnection` (`src/ui/pages/StreamPage.cpp:1701`), but the handler's `m_pendingPlay.valid` guard conflicts with `onSourceActivated()` clearing that flag before playback (`src/ui/pages/StreamPage.cpp:1544`, `src/ui/pages/StreamPage.cpp:1500`).

### Per-session signal connections

- `VideoPlayer::progressUpdated`, `closeRequested`, and `streamNextEpisodeRequested` are disconnected at the top of `StreamPage::onReadyToPlay()` (`src/ui/pages/StreamPage.cpp:1633`) and again on close/failure/streamStopped (`src/ui/pages/StreamPage.cpp:1708`, `src/ui/pages/StreamPage.cpp:1901`, `src/ui/pages/StreamPage.cpp:1919`).
- The Shift+N member-slot connection uses `Qt::UniqueConnection` (`src/ui/pages/StreamPage.cpp:1701`). Qt documents that duplicate ordinary connections emit duplicate calls, and that `Qt::UniqueConnection` prevents duplicate member-function connections; it does not apply to lambda/functor connections. Source: https://doc.qt.io/qt-6/qobject.html#connect and https://doc.qt.io/qt-6/qt.html#ConnectionType-enum.
- The `closeRequested` StreamPage lambda is not `UniqueConnection`, but the preceding wildcard disconnect removes prior StreamPage receivers for that signal (`src/ui/pages/StreamPage.cpp:1635`). MainWindow's persistent close connection remains (`src/ui/MainWindow.cpp:127`), so player close always has at least two receivers while Stream mode playback is active.
- `StreamAggregator::streamsReady` is connected per request and disconnected using wildcard receiver disconnects (`src/ui/pages/StreamPage.cpp:1152`, `src/ui/pages/StreamPage.cpp:1308`). This prevents duplicate fires for the same sender/receiver pair, but because the same aggregator object is reused for normal source picker and next-episode prefetch, one flow can clear another flow's receiver.
- `SidecarProcess` connects to QProcess and emits typed Qt signals persistently at construction (`src/ui/player/SidecarProcess.cpp:80`). `processLine()` reads `name` and `payload`, but does not compare the incoming JSON `sessionId` with `m_sessionId` before emitting (`src/ui/player/SidecarProcess.cpp:339`).

### Sidecar process lifecycle

- `VideoPlayer::openFile()` calls `resetFileState()` then `stopPlayback()` before setting the new file and pending open (`src/ui/player/VideoPlayer.cpp:276`, `src/ui/player/VideoPlayer.cpp:282`).
- `stopPlayback()` sends `stop` and immediately sends `shutdown` if the process is running (`src/ui/player/VideoPlayer.cpp:369`). It does not wait for `closed`, `finished`, or `processClosed`, and it does not clear `m_pendingFile`.
- After `stopPlayback()`, the same `openFile()` call checks `m_sidecar->isRunning()`. If the old process has not exited yet, it sends `open` directly to that process (`src/ui/player/VideoPlayer.cpp:342`). This matches the prior log pattern in `agents/chat.md:18535`: second `openFile` sent `stop`, `shutdown`, then `open` before the sidecar reported `closed`.
- `SidecarProcess::sendOpen()` generates a new `m_sessionId` for every open (`src/ui/player/SidecarProcess.cpp:138`). `sendStop()` and `sendShutdown()` use the current session id (`src/ui/player/SidecarProcess.cpp:151`).
- Native sidecar allows `open` and `shutdown` to bypass the command session guard (`native_sidecar/src/state_machine.h:24`), while other commands are dropped if their session differs from `g_state.sessionId()` (`native_sidecar/src/main.cpp:1266`).
- Native events carry a `sessionId` (`native_sidecar/src/protocol.cpp:106`), and decoder callbacks capture the open worker's `sid` (`native_sidecar/src/main.cpp:367`). The Qt side currently discards that field (`src/ui/player/SidecarProcess.cpp:350`).

### HTTP server per-connection lifecycle

- `handleConnection()` copies the registry entry once after parsing the URL (`src/core/stream/StreamHttpServer.cpp:179`). If `StreamEngine::stopStream()` unregisters the file later (`src/core/stream/StreamEngine.cpp:296`), the worker still has its local `FileEntry`.
- `waitForPieces()` polls `TorrentEngine::haveContiguousBytes()` every 200ms for up to 15s (`src/core/stream/StreamHttpServer.cpp:81`). It does not accept `StreamHttpServer*` or a cancellation token, so it does not notice `m_shuttingDown` or per-stream removal until it returns.
- `StreamHttpServer::stop()` drains active connections for 2s (`src/core/stream/StreamHttpServer.cpp:387`), but normal stream close calls `StreamEngine::stopStream()`, not HTTP server `stop()` (`src/core/stream/StreamEngine.cpp:279`).
- If a worker is sleeping in `waitForPieces()` when `stopStream()` removes the torrent (`src/core/stream/StreamEngine.cpp:300`), the next `haveContiguousBytes()` result depends on `TorrentEngine`'s unknown-torrent behavior. From StreamHttpServer's perspective, the observed behavior is bounded only by the 15s wait timeout, not by session end.
- The 30s idle watchdog is local stack state inside `handleConnection()` (`src/core/stream/StreamHttpServer.cpp:251`) and does not hold external references beyond the copied `FileEntry`, server pointer, engine pointer, and local socket/file.

### QSettings and persistent state

- `_currentEpKey` is a dynamic QObject property and is not persisted. It bridges the current stream session into the progress lambda (`src/ui/pages/StreamPage.cpp:1643`) and the resume lookup (`src/ui/pages/StreamPage.cpp:1744`).
- Stream progress writes happen from `VideoPlayer::saveProgress()` emitting `progressUpdated()` every sidecar time update (`src/ui/player/VideoPlayer.cpp:1965`), then StreamPage's current progress connection writes to `"stream"` using the current `_currentEpKey` (`src/ui/pages/StreamPage.cpp:1641`). If old sidecar events are emitted after a new `_currentEpKey` is installed, the write target is the new key.
- VideoPlayer's own `"videos"` progress writes are gated by `PersistenceMode::LibraryVideos` (`src/ui/player/VideoPlayer.cpp:1979`). StreamPage sets `PersistenceMode::None` before stream open and resets it on close/failure/stopped (`src/ui/pages/StreamPage.cpp:1733`, `src/ui/pages/StreamPage.cpp:1710`, `src/ui/pages/StreamPage.cpp:1904`, `src/ui/pages/StreamPage.cpp:1922`).
- Stream source choices and series choices are intentionally persistent via `StreamChoices::saveChoice()` and `saveSeriesChoice()` (`src/ui/pages/StreamPage.cpp:1575`, `src/ui/pages/StreamPage.cpp:1584`). They are not per-session leak candidates by themselves.

## Reference behavior

- WebTorrent documents teardown as callback-gated: `client.remove()` / `torrent.destroy()` callbacks mean open sockets are closed and storage is closed/destroyed before completion; `client.destroy()` also has a graceful-close callback. Source: https://webtorrent.io/docs.
- WebTorrent's HTTP server model creates stream URLs backed by torrent-aware reads; file streams prioritize pieces needed by the stream, and example cleanup closes the server and destroys the client. Source: https://webtorrent.io/docs.
- Peerflix handles HTTP range requests by serving `file.createReadStream(range)` from torrent-stream rather than exposing process-global player state to the decoder. Source: https://github.com/mafintosh/peerflix/blob/1b580420671677650ce4fd7d37daeddb58350b22/index.js#L150-L168.
- mpv's lavf demuxer installs custom read/seek callbacks and interrupt/io hooks around libavformat, so cancellation and teardown are owned by a stream/session abstraction instead of inferred from stale process state. Source: https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L292-L349 and https://github.com/mpv-player/mpv/blob/da4789c2ddce92713347ecf1b5912fceb812ea39/demux/demux_lavf.c#L1033-L1083.
- Qt documents that duplicate connections emit duplicate calls, `Qt::UniqueConnection` prevents exact member-slot duplicates, queued events may still be delivered after disconnect, and lambda/functor connections require care for captured object lifetime. Source: https://doc.qt.io/qt-6/qobject.html#connect.

## Gaps (ranked P0 / P1 / P2)

**P0 (highest likelihood of intermittent user-visible failure):**

- Source-switch teardown is reentrant and can clear the new session while `startStream()` is still starting it. Observed: `onSourceActivated()` installs the new `_currentEpKey` and UI state before calling controller start (`src/ui/pages/StreamPage.cpp:1587`, `src/ui/pages/StreamPage.cpp:1610`); controller start synchronously calls `stopStream()` (`src/ui/pages/stream/StreamPlayerController.cpp:26`); old `stopStream()` emits `streamStopped()` (`src/ui/pages/stream/StreamPlayerController.cpp:81`); StreamPage handles it by clearing `_currentEpKey`, resetting player connections/persistence, and calling `showBrowse()` unless the next-episode overlay is visible (`src/ui/pages/StreamPage.cpp:1915`, `src/ui/pages/StreamPage.cpp:1932`). Reference: WebTorrent teardown uses callback-gated removal/destroy completion before the next lifecycle step (https://webtorrent.io/docs). Impact: high likelihood for user-visible intermittent failure on source-switch-mid-play or any future start-while-active route, because old teardown mutates new-session state.
- `VideoPlayer::openFile()` can send `stop+shutdown` and then `open` to the same still-running sidecar process. Observed: open always calls `stopPlayback()` first (`src/ui/player/VideoPlayer.cpp:282`), stop sends `stop` and `shutdown` without waiting (`src/ui/player/VideoPlayer.cpp:369`), then open sends `open` directly if `isRunning()` still returns true (`src/ui/player/VideoPlayer.cpp:342`). The prior incident log shows this exact ordering in `agents/chat.md:18535`. Reference: WebTorrent's remove/destroy callbacks are documented as completion points after sockets/storage are closed (https://webtorrent.io/docs). Impact: very high likelihood; this is the same class as the just-fixed double-open blank-player race, but the vulnerable boundary still exists in VideoPlayer.
- `SidecarProcess` ignores incoming event `sessionId`, so stale sidecar events can mutate the current session. Observed: `sendOpen()` creates a new session id (`src/ui/player/SidecarProcess.cpp:138`), native events carry the open worker sid (`native_sidecar/src/main.cpp:367`), but `processLine()` dispatches by event name and payload without checking `obj["sessionId"]` (`src/ui/player/SidecarProcess.cpp:339`). Reference: native sidecar already treats session id as a guard for commands (`native_sidecar/src/main.cpp:1266`); Qt docs warn queued signal events can still be delivered after disconnect (https://doc.qt.io/qt-6/qobject.html#connect). Impact: high likelihood because stale `probe_complete`, `first_frame`, `tracks_changed`, or `time_update` can rewrite VideoPlayer fields and StreamPage progress under the new `_currentEpKey`.

**P1 (probable under failed-open, weak-swarm, or rapid navigation cycles):**

- `StreamPlayerController::m_infoHash` can survive failure into the next start. Observed: start resets ids/counters but not `m_infoHash` (`src/ui/pages/stream/StreamPlayerController.cpp:28`); poll sets it only if empty (`src/ui/pages/stream/StreamPlayerController.cpp:101`); timeout/engine-error set `m_active=false` without clearing it (`src/ui/pages/stream/StreamPlayerController.cpp:91`, `src/ui/pages/stream/StreamPlayerController.cpp:152`); the next start's initial `stopStream()` returns early if inactive (`src/ui/pages/stream/StreamPlayerController.cpp:65`). Impact: likely under failed-open/reopen cycles; stale `currentInfoHash()` feeds seek pre-gate, deadline retargeting, and stopStream routing.
- The old failure timer can navigate after a new user action because it has no session generation. Observed: `onStreamFailed()` schedules a 3s `singleShot` that calls `showBrowse()` when the controller is inactive (`src/ui/pages/StreamPage.cpp:1907`). It does not capture `_currentEpKey`, infoHash, or a session generation. Impact: likely when a failed stream is followed by browsing or direct-url activity before the 3s timer fires.
- Active HTTP workers are not cancelled by per-stream teardown. Observed: per-stream stop removes the stream record, unregisters the file, clears deadlines, and removes the torrent (`src/core/stream/StreamEngine.cpp:286`), while an existing worker can remain inside `waitForPieces()` for up to 15s with a copied `FileEntry` (`src/core/stream/StreamHttpServer.cpp:179`, `src/core/stream/StreamHttpServer.cpp:81`). Impact: likely to cause delayed file/socket release and retry noise on close/reopen under weak swarm conditions; crash likelihood depends on TorrentEngine's unknown-torrent behavior.
- `StreamEngine::streamError` is emitted but not observed by the inspected controller/page path. Observed: `StreamEngine` emits for no-video and torrent errors (`src/core/stream/StreamEngine.cpp:517`, `src/core/stream/StreamEngine.cpp:620`), but inspected source search found no connection to `StreamEngine::streamError`. Impact: medium likelihood; some stream records may sit in `m_streams` until controller timeout or explicit user stop, preserving deadlines/registry/torrent state longer than intended.
- VideoPlayer stop does not clear file identity state. Observed: `stopPlayback()` stops timers/canvas/sidecar and track arrays but leaves `m_currentFile`, `m_pendingFile`, `m_pendingStartSec`, playlist, and last-known crash recovery identity intact (`src/ui/player/VideoPlayer.cpp:358`). Impact: medium likelihood; most normal stop paths rely on intentional sidecar shutdown to suppress recovery, but any unexpected process event after close still sees a current file.

**P2 (lower likelihood or visible but not the close/reopen class):**

- `m_lastDeadlineUpdateMs` is session-independent. Observed: it is only initialized in the header and updated in the progress lambda (`src/ui/pages/StreamPage.h:310`, `src/ui/pages/StreamPage.cpp:1658`). Impact: lower likelihood; fast close/reopen can delay the first sliding-window deadline update for the new stream by up to 2s.
- `onSourceActivated()` uses partial next-episode cleanup. Observed: it resets `m_nearEndCrossed` and `m_nextPrefetch` directly (`src/ui/pages/StreamPage.cpp:1547`) instead of calling `resetNextEpisodePrefetch()`, so it does not clear `m_nextShortcutPending` or disconnect prefetch aggregator receivers. Impact: lower current likelihood because Shift+N appears gated off by `m_pendingPlay.valid`, but this is the same inconsistent-cleanup pattern.
- `m_pendingPlay.valid` is not the playback-session identity it is later treated as. Observed: it is cleared at source activation (`src/ui/pages/StreamPage.cpp:1544`), but Shift+N requires it to remain valid during playback (`src/ui/pages/StreamPage.cpp:1500`). Impact: user-visible shortcut miss, not the primary intermittent close/reopen class.

## Recommended follow-ups (advisory)

- Consider a single StreamPage/Controller `resetSessionState(reason)` or `PlaybackSession` object that owns `_currentEpKey`, pending play, next-episode state, seek retry state, deadline timestamp, and StreamPage-owned player connections. The audit found the same fields reset through several partial paths instead of one session boundary.
- Consider giving every async closure/timer a monotonic playback generation. The existing `m_seekRetryState` fix is the local version of this pattern; the 3s failure timer, aggregator lambdas, and progress lambda do not share it.
- Investigate a sidecar open/stop fence: either wait for `processClosed`/`finished` after shutdown before sending a new `open`, or avoid `shutdown` for same-process file switches and use a stop/open protocol with an explicit ack. The current `isRunning()` branch is not a completion barrier.
- Investigate filtering all `SidecarProcess::processLine()` events by `sessionId`, with explicit exceptions only for process-global events such as `ready`/`closed` if needed.
- Consider splitting "stop old stream because user is ending playback" from "stop old stream as part of replacing it." Today both paths emit the same `streamStopped()` and StreamPage treats it as a user-facing end.
- Consider adding per-stream cancellation to `StreamHttpServer::waitForPieces()` so unregister/remove-torrent breaks active waits promptly rather than waiting up to 15s.
- Consider replacing wildcard disconnects for shared aggregators with stored `QMetaObject::Connection` handles per flow. Qt's own docs recommend storing handles to avoid mismatched or overly broad disconnects (https://doc.qt.io/qt-6/qobject.html#connect).
