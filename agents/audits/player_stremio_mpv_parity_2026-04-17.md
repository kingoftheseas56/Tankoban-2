# Audit — Video Player + sidecar Stremio/mpv parity — 2026-04-17

By Agent 7 (Codex). For Video Player + sidecar / Agent 3.
Reference comparison: Tankoban current `src/` + `native_sidecar/`, local Stremio reference sources, local mpv source/manual, `C:\tools\mpv\mpv.exe`, and official mpv/Stremio web documentation.
Scope: This audit compares Tankoban's current in-app video player and native sidecar against Stremio/mpv-style streaming playback behavior, especially stream-mode readiness, buffering, seeking, tracks/subtitles, HDR, timing, and observable player state. I did not build or run Tankoban; AGENTS.md forbids Agent 7 from compiling or executing the project. The only executable probe was `C:\tools\mpv\mpv.exe --version`, `--list-properties`, and `--list-options` against the provided reference binary.

## Observed behavior (in our codebase)

- Tankoban's Qt-to-sidecar IPC is a JSON-line command/event protocol. `SidecarProcess::sendCommand()` emits `{type:"cmd", name, sessionId, seq, payload}`, and `sendOpen()` regenerates the session id while preserving HTTP URLs instead of converting them to native path separators. Citation: `src/ui/player/SidecarProcess.cpp:124-153`.
- The Qt player exposes a fixed command set: open, pause/resume, seek, frame step, stop, volume/mute, rate, tracks, subtitle visibility/delay/style, external subtitle load, filters, tone mapping, zero-copy, canvas sizing, resize, and shutdown. Citation: `src/ui/player/SidecarProcess.h:42-64`.
- The Qt player exposes a fixed event/signal set: ready, first frame, time update, state, tracks, EOF, error, subtitle text/visibility/delay, filters, media info, frame step, D3D11 texture, overlay SHM, and buffering start/end. There is no signal for demuxer cache ranges, cache percentage, byte/input rate, seekability, or paused-for-cache. Citation: `src/ui/player/SidecarProcess.h:146-181`.
- Event intake filters stale session-scoped events by session id, then dispatches known event names. HTTP stall events map only to `bufferingStarted()`, and `playing` maps to `bufferingEnded()`. Citation: `src/ui/player/SidecarProcess.cpp:397-560`.
- The seek slider stores duration and chapter markers, paints chapter ticks, and maps mouse position to direct seeks. It does not expose a buffered-range/cache API or paint multiple seekable ranges. Citation: `src/ui/player/SeekSlider.cpp:52-96` and `src/ui/player/SeekSlider.cpp:116-151`.
- The loading overlay has two visible states: `Loading — <basename>` and `Buffering…`. It is driven by player open/idle, sidecar buffering start/end, and first-frame dismissal. Citation: `src/ui/player/LoadingOverlay.cpp:22-52`, `src/ui/player/LoadingOverlay.cpp:120-127`, and `src/ui/player/VideoPlayer.cpp:1380-1398`.
- The sidecar probe path uses FFmpeg HTTP options for stream URLs: reconnect enabled, 60s connect timeout, 30s read timeout, 20 MB probe size, and 10s analyze duration. It also extracts duration, audio/subtitle tracks, chapter metadata, color metadata, and HDR side data. Citation: `native_sidecar/src/demuxer.cpp:39-68` and `native_sidecar/src/demuxer.cpp:189-231`.
- The video decoder path uses similar HTTP options, enables D3D11VA when possible, creates a shared D3D11 presenter path, and allocates subtitle overlay SHM when the presenter is ready. Citation: `native_sidecar/src/video_decoder.cpp:212-223` and `native_sidecar/src/video_decoder.cpp:286-390`.
- On HTTP read stalls, the video decoder emits `buffering`, retries every 500 ms, emits `STREAM_TIMEOUT:no data for 30 seconds` after 60 stalls, and emits `playing` when a later read succeeds. Citation: `native_sidecar/src/video_decoder.cpp:1077-1123`.
- Video seek handling in the decoder uses `av_seek_frame(fmt_ctx, -1, ts, AVSEEK_FLAG_BACKWARD)`, flushes codec buffers, logs the target, and continues. I did not find an adjacent decode-to-target/high-resolution seek loop in the inspected video seek block. Citation: `native_sidecar/src/video_decoder.cpp:1061-1075`.
- The sidecar emits tracks and media info before first frame. Track payloads include default/forced flags and audio channels/sample rate; media info carries HDR/color fields, chapters, and audio device info. Citation: `native_sidecar/src/main.cpp:359-423`.
- Track switching restarts the audio decoder when the audio stream changes, switches subtitle stream ids, preloads subtitle packets for the selected embedded subtitle, and emits updated active ids. Citation: `native_sidecar/src/main.cpp:1004-1086`.
- Subtitles use libass with DirectWrite on Windows, extract embedded fonts, support ASS/SSA, SRT/SubRip/mov_text/text, PGS/DVD bitmap subtitles, and allow renderer-level font scaling/margin/outline state. The outline toggle explicitly does not override ASS/SSA file styles and takes effect for text-subtitle header rebuilds on the next load. Citation: `native_sidecar/src/subtitle_renderer.cpp:138-159`, `native_sidecar/src/subtitle_renderer.cpp:335-411`, and `native_sidecar/src/subtitle_renderer.cpp:648-685`.
- The UI sends playback speed changes through `sendSetRate()` from chips, shortcuts, and context menu. The sidecar `set_rate` handler clamps 0.0625-16.0 and calls `g_clock.set_rate(rate)`. Audio speed compensation is a separate 500 ms sync ticker path whose sidecar handler is named `set_audio_speed`. Citation: `src/ui/player/VideoPlayer.cpp:1248-1268`, `native_sidecar/src/main.cpp:943-949`, and `native_sidecar/src/main.cpp:1429-1472`.
- HDR and chapters are forwarded from `media_info` into `FrameCanvas`/seek UI; D3D11 shared texture and overlay SHM are attached in the player. Citation: `src/ui/player/VideoPlayer.cpp:1618-1648` and `src/ui/player/VideoPlayer.cpp:1707-1732`.

## Streaming-mode-specific player behaviors vs library-file playback

- Tankoban's player entry point treats both local files and stream URLs as `openFile(...)` inputs, then sends the final path to the sidecar. The IPC keeps HTTP URLs unchanged while converting non-HTTP paths to native separators. Citation: `src/ui/player/VideoPlayer.h:49-52` and `src/ui/player/SidecarProcess.cpp:140-153`.
- Stream playback is explicitly protected from local-library progress pollution by `VideoPlayer::PersistenceMode`: stream URLs can run in a no-persistence/current-session mode, while local files can use file/video progress persistence. Citation: `src/ui/player/VideoPlayer.h:63-85`.
- On the native sidecar side, HTTP/HTTPS inputs receive stream-specific FFmpeg options on probe and decode paths: reconnect behavior, connect/read timeouts, larger probe size, and longer analyze duration. Local paths skip those HTTP option blocks. Citation: `native_sidecar/src/demuxer.cpp:39-68` and `native_sidecar/src/video_decoder.cpp:212-223`.
- Once opened, local files and stream URLs share the same main player UI surfaces: time updates, duration, track payloads, media info, chapters, seek bar, loading overlay, speed controls, subtitles, and D3D11/SHM presentation. Citation: `src/ui/player/SidecarProcess.h:146-181`, `src/ui/player/VideoPlayer.cpp:1380-1398`, `src/ui/player/VideoPlayer.cpp:1618-1648`, and `src/ui/player/VideoPlayer.cpp:1707-1732`.
- Stream-specific runtime behavior currently appears at the stall/recovery boundary: HTTP read failures emit `buffering`, successful later reads emit `playing`, and the UI maps those events to a `Buffering…` overlay. Local files do not enter that HTTP stall branch. Citation: `native_sidecar/src/video_decoder.cpp:1077-1123`, `src/ui/player/SidecarProcess.cpp:544-560`, and `src/ui/player/LoadingOverlay.cpp:35-52`.
- I did not find a player-layer distinction that exposes stream-specific buffered/seekable ranges, torrent/server statistics, mpv-style cache state, or Stremio-style `stream_state` to the seek UI. Citation: `src/ui/player/SidecarProcess.h:146-181`, `src/ui/player/SeekSlider.cpp:52-96`, and `C:\Users\Suprabha\Downloads\Stremio Reference\stremio-core-development\stremio-core-development\stremio-core-web\src\model\serialize_player.rs:107-119`.

## Gaps (ranked P0 / P1 / P2)

**P0 (user-blocking or severely degrading):**

- Stream-mode buffered/seekable state is not surfaced to the Qt player. Observed: Tankoban exposes `bufferingStarted()`/`bufferingEnded()` but not seekable ranges, buffered ranges, cache percent, cache duration, reader PTS, raw input rate, or paused-for-cache; the seek slider paints only progress and chapter ticks. Citation: `src/ui/player/SidecarProcess.h:146-181`, `src/ui/player/SidecarProcess.cpp:544-560`, and `src/ui/player/SeekSlider.cpp:52-96`. Reference: mpv documents `demuxer-cache-state.seekable-ranges`, `cache-buffering-state`, and `paused-for-cache`; Stremio HTML video exposes both `buffering` and `buffered`. Citation: `C:\Users\Suprabha\Downloads\Stremio Reference\mpv-master\mpv-master\DOCS\man\input.rst:2588-2692` and `C:\Users\Suprabha\Downloads\Stremio Reference\stremio-video-master\stremio-video-master\src\HTMLVideo\HTMLVideo.js:162-181`. Impact: for torrent-backed streams, the UI cannot distinguish "duration exists" from "target time is actually cached/seekable", so seek UX can look local-file-complete while the backend is still piece-gated.
- Rebuffering is modeled as an error/retry side effect, not as a first-class cache-pause state machine. Observed: on HTTP stalls the decoder emits `buffering`, retries every 500 ms, and times out after 30 seconds; the Qt overlay shows `Buffering…` but receives no cache-fill progress or resume threshold. Citation: `native_sidecar/src/video_decoder.cpp:1077-1123`, `src/ui/player/LoadingOverlay.cpp:35-52`, and `src/ui/player/VideoPlayer.cpp:1389-1398`. Reference: mpv documents automatic cache pause, cache-pause wait, initial cache pause, and cache-pause behavior after seeking. Citation: `C:\Users\Suprabha\Downloads\Stremio Reference\mpv-master\mpv-master\DOCS\man\options.rst:5458-5485`. Impact: a stream can stall and recover, but the app cannot present mpv-like "paused for cache" semantics or explain when playback is likely to resume.
- The current sidecar API is a fixed event surface rather than a queryable/observable property graph. Observed: Qt dispatches only recognized event names and has fixed signals; Stremio's video layer and mpv both rely heavily on property observation (`propValue`/`propChanged` in Stremio, runtime properties in mpv). Citation: `src/ui/player/SidecarProcess.cpp:397-560`, `C:\Users\Suprabha\Downloads\Stremio Reference\stremio-video-master\stremio-video-master\src\StremioVideo\StremioVideo.js:47-78`, and `C:\Users\Suprabha\Downloads\Stremio Reference\mpv-master\mpv-master\DOCS\man\input.rst:2181-2208`. Impact: parity work has to add bespoke commands/events for each capability, while mpv/Stremio-like implementations can observe a richer state model consistently across playback modes.

## Hypothesized root causes (Agent 3 to validate)

- **Hypothesis —** The player-side IPC grew around specific sidecar commands/events rather than a queryable property model, so stream-mode cache, seekability, and readiness data have no stable transport shape yet. **Agent 3 to validate.**
- **Hypothesis —** The current stream buffering model is inferred from FFmpeg read failures and recoveries, so readiness is discovered after the decoder stalls rather than represented before the user seeks or playback underruns. **Agent 3 to validate.**
- **Hypothesis —** Tankoban's custom sidecar has implemented many isolated mpv-like features, but the largest remaining parity gaps sit at integration boundaries: demuxer cache state, seek semantics, track/subtitle property state, and playback timing. **Agent 3 to validate.**
- **Hypothesis —** The shared local-file and stream-file UI path hides stream-specific state because the player was built to treat HTTP URLs as playable paths, while Stremio separates stream conversion/server state from the inner video implementation. **Agent 3 to validate.**
- **Hypothesis —** Playback speed may be split between a master clock rate path and an A/V drift correction path, leaving non-1.0x audio behavior dependent on interactions outside the inspected `set_rate` handler. **Agent 3 to validate.**

## Recommended follow-ups (advisory)

- Consider whether the next player objective should define a minimal "observable playback state" contract before adding more individual UI features. Candidate comparison fields: `buffering`, `buffered/seekable ranges`, `pausedForCache`, `cachePercent`, `rawInputRate`, `demuxerViaNetwork`, `seekable`, `partiallySeekable`, `tracks`, `chapters`, and `streamState`.
- Consider a focused stream-mode reproduction matrix for local file vs HTTP stream vs torrent-backed HTTP stream: open latency, first-frame latency, seek into cached range, seek outside cached range, rebuffer after seek, stall recovery, and timeout messaging.
- Investigate whether `av_seek_frame(... AVSEEK_FLAG_BACKWARD)` is intentionally "good enough" or whether exact seek parity is expected for chapter jumps, resume, subtitle sync, and frame step.
- Investigate non-1.0x playback with audible audio: 0.5x, 0.75x, 1.25x, 1.5x, and 2.0x, with and without subtitles, to verify whether clock rate, audio decoder compensation, pitch, and subtitle timing remain coherent.
- Consider whether stream-specific sidecar telemetry should be visible in developer/debug UI before being exposed as user-facing controls, so Agent 3 can compare Tankoban sessions directly against mpv `demuxer-cache-state` and Stremio stream-state behavior.
